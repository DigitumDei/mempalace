## Executive Summary

This branch establishes the Phase 1 Rust workspace for mempalace-rs, introducing the crate layout, core domain types, configuration loading, path resolution, embedding profile selection, shared tracing, and a minimal CLI entrypoint. Two blocking defects were identified with full consensus across all reviewers: `ConfigFileV1.version` has no `#[serde(default)]` attribute, causing a deserialization failure on any Python-era `config.json` that lacks the field; and `base_dir_override` is silently ignored when `config.json` is absent because `ConfigFileV1::default()` injects a hardcoded `palace_path` that wins the merge chain in `load_from_sources`. Both defects are directly traceable in the code and are not caught by the existing test suite. A third dispute â€” whether `ProjectConfig` silently dropping `rooms` is a Phase 1 blocking regression or only a forward-looking non-blocking gap â€” was not resolved by the panel. The verdict is revise to address the two confirmed blockers before this branch is merged.

## Blocking Issues

- `ConfigFileV1.version` missing `#[serde(default)]` breaks legacy config deserialization (mempalace-rs/crates/mempalace-config/src/config.rs): All reviewers confirmed that `version: u32` at `config.rs:23` carries no serde default attribute. Any `config.json` produced by the Python implementation that omits the `version` key â€” which is the normal Python-era shape â€” will fail with `missing field 'version'` on first Rust load, making this a crash-on-upgrade path for all existing users.
- `base_dir_override` silently ignored when `config.json` is absent (mempalace-rs/crates/mempalace-config/src/config.rs): All reviewers confirmed that `read_config_file` returns `ConfigFileV1::default()` when the file does not exist, and that default sets `palace_path: Some("~/.mempalace/palace")`. In `load_from_sources`, the `.or(file.palace_path)` arm picks up this hardcoded default before the `paths.palace_dir` fallback, so passing `base_dir_override = Some("/custom/dir")` silently resolves to the wrong palace path. The existing test suite never surfaces this because `init_and_load_defaults_round_trip` always calls `init_default` before loading.

## Non-Blocking Issues

- `serde_yaml = "0.9.34+deprecated"` introduces day-one technical debt (mempalace-rs/Cargo.toml): The crate author has officially abandoned serde_yaml and recommends alternatives. Using it in a new workspace starts a deprecation clock immediately.
- CLI logs 'foundation initialized' without calling `init_default` (mempalace-rs/crates/mempalace-cli/src/main.rs): `main.rs` calls `load_with_env(None)` only; it never invokes `init_default`, so no on-disk foundation state is established despite the log message.
- `ProjectConfig` silently discards `rooms`, which the Python miner requires for routing (mempalace-rs/crates/mempalace-config/src/config.rs): The committed fixture at `tests/fixtures/phase0/inputs/project_alpha/mempalace.yaml` contains a `rooms` array that Python's `detect_room` depends on. The Rust struct only models `wing`, silently dropping `rooms` via serde's default unknown-field behaviour. Classified non-blocking because Phase 1 scope documents do not commit to room-routing parity, and no Rust consumer currently reads `rooms`; however, this must be addressed before any Phase 2 room-routing work begins.
- No fallback to legacy `mempal.yaml` for project config (mempalace-rs/crates/mempalace-config/src/config.rs): Python's `miner.py` falls back to `mempal.yaml` when `mempalace.yaml` is absent. `load_project_config` in Rust hardcodes only the new filename, leaving a real compatibility gap for projects that have not yet migrated their config filename.
- `init_default` does not create `palace_dir` (mempalace-rs/crates/mempalace-config/src/config.rs): `fs::create_dir_all` is called only for `paths.base_dir`. The `palace_dir` path is written into the config JSON as the default storage root but is never materialised, causing downstream write failures. The Python baseline also defers storage-dir creation, so this may be intentional, but it is undocumented.
- `tracing-subscriber` pulled into the `mempalace-core` library crate (mempalace-rs/crates/mempalace-core/Cargo.toml): Library crates should only instrument with `tracing`; subscriber initialisation and the resulting `regex` dependency belong in the binary (`mempalace-cli`) or a dedicated init crate. Every downstream consumer of `mempalace-core` links the subscriber stack unnecessarily.
- Undocumented `MEMPAL_PALACE_PATH` env var alias (mempalace-rs/crates/mempalace-config/src/config.rs): The fallback to `MEMPAL_PALACE_PATH` is intentional Python legacy compatibility but carries no comment pointing to that rationale. Future readers will question or silently remove it.
