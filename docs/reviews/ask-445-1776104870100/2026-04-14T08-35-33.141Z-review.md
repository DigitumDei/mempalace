## Executive Summary

This branch completes Phase 3 of the Rust embeddings subsystem by wiring in a fastembed-based provider with profile mappings for both balanced and low_cpu modes, adding startup/cache validation logic, a benchmarking path with CI publication of warm baselines, and closing out the associated task tracking. The core implementation is structurally sound — benchmark warmup exclusion is correct, cache-status classification logic is correctly ordered, and the model cache path is derived from fastembed's own model code — but the validation layer's reliability guarantee is broken in two distinct ways: `resolve_model_files_root` falls back to an arbitrary snapshot directory when `refs/main` is missing or stale, meaning `startup_validation()` can return `Ready` for a cache that `hf-hub` cannot actually resolve offline; and if `fastembed 5.13.2`'s `pull_from_hf()` prefers `HF_HOME` over the explicitly-passed `cache_dir`, validation and runtime can inspect entirely different caches, producing false Ready or false MissingAssets results. Because offline determinism is a stated Phase 3 exit criterion, these correctness defects in the validation path are blocking.

## Blocking Issues

- `resolve_model_files_root` fallback silently reports Ready for caches hf-hub cannot use (mempalace-rs/crates/mempalace-embeddings/src/lib.rs): When `refs/main` is absent or names a revision whose snapshot directory does not exist, the function falls back to the alphabetically-first directory under `snapshots/`. `hf-hub` resolves models exclusively via `refs/main` → `snapshots/<revision>`, so `startup_validation()` can return `StartupValidationStatus::Ready` while `TextEmbedding::try_new` subsequently fails at the network or resolution layer rather than producing the deterministic offline error the validation layer is designed to surface. Both the missing-ref and stale-ref sub-cases produce this false-positive path. No test covers either sub-case.
- `HF_HOME` environment variable can cause validation and runtime to inspect different cache roots (mempalace-rs/crates/mempalace-embeddings/src/lib.rs): Validation inspects `self.config.cache_root`, but `fastembed 5.13.2`'s `pull_from_hf()` is reported to prefer `HF_HOME` over the explicitly-passed cache directory. If this precedence is in effect, the validation layer can classify a cache as Ready while the runtime uses a different directory, or classify a cache as MissingAssets when the runtime-preferred cache is actually complete. This would break the entire offline-determinism guarantee for users with `HF_HOME` set. No test covers this path.

## Non-Blocking Issues

- CLI `init` emits 'mempalace cli foundation initialized' regardless of validation status (mempalace-rs/crates/mempalace-cli/src/main.rs): `startup_validation()` is called and logged, but the result is never branched on. An operator doing a first-time init with no cached model assets receives the same affirmative success log as a healthy warm cache, with no actionable indication that downloads are required before the provider can be used.
- Low-CPU RSS budget constants contradict Phase 11 spec by more than 50% (mempalace-rs/crates/mempalace-embeddings/src/lib.rs): `from_profile(EmbeddingProfile::LowCpu)` hardcodes `low_cpu_idle_rss_budget_mb: Some(700)` and `low_cpu_ingest_rss_budget_mb: Some(1_200)`, while `docs/RustMigrationTasks.md` documents Phase 11 gates as 450 MB and 850 MB. These fields are dead code in Phase 3, but Phase 11 documentation explicitly points back to this layer; if consumed as-is, the enforcement gates are pre-broken.
- No `timeout-minutes` on `embedding-baselines` CI job (.github/workflows/mempalace-rs-storage.yml): The job sets `MEMPALACE_EMBED_ALLOW_DOWNLOADS: "1"` and downloads multi-gigabyte ONNX model assets. A rate-limit, CDN stall, or slow runner will consume GitHub Actions' full 6-hour default timeout with no bound. A `timeout-minutes: 30` at the job level would limit the blast radius.
- `fastembed` default features introduce undocumented `native-tls`/OpenSSL host dependency (mempalace-rs/crates/mempalace-embeddings/Cargo.toml): Adding `fastembed = "5.13.2"` with default features transitively pulls in `native-tls` and `openssl-sys`. `cargo check -p mempalace-cli --locked` fails on hosts without system OpenSSL headers. GitHub runners typically have this, but the new host requirement is not documented in the workflow or contributing guide, creating friction for local and container builds.
- Corrupted cache + `allow_downloads = true` passes through to `try_new` without clearing (mempalace-rs/crates/mempalace-embeddings/src/lib.rs): The guard `!validation.is_ready() && !self.config.allow_downloads` passes when downloads are allowed regardless of whether the status is `MissingAssets` or `CorruptedCache`. When the cache is `CorruptedCache`, `try_new` is called against the corrupted directory without first purging it. Whether fastembed re-downloads over corrupted assets or errors is not guaranteed by this layer's contracts.
