## Executive Summary

This branch completes Phase 3 of the Rust embeddings subsystem by activating fastembed as the live backend, mapping both `balanced` and `low_cpu` profiles to real runtime configurations, adding a startup/cache validator, and publishing warm-run benchmarks via CI. Two correctness defects in the startup validator undermine the branch's primary deliverable: `discover_cache_assets` recursively scans the entire shared cache root rather than the selected model's subdirectory, so any previously cached model causes a false `Ready` status for a different profile and renders the `allow_downloads = false` offline guard unenforceable; additionally, the validator checks only `onnx`, `tokenizer.json`, and `config.json` while fastembed also requires `special_tokens_map.json` and `tokenizer_config.json`, producing false positives on partial caches. Because the startup validator is an explicit Phase 3 exit criterion and both bugs make its readiness signal unreliable, the branch is not ready to merge.

## Blocking Issues

- Cache validation not scoped to selected model â€” offline guard becomes unenforceable (mempalace-rs/crates/mempalace-embeddings/src/lib.rs): `discover_cache_assets` walks the full `cache_root` recursively and tallies matching files across all subdirectories. fastembed stores each model in a named subdirectory (e.g., `fast-all-MiniLM-L6-v2` vs `fast-all-MiniLM-L6-v2-q`). If the balanced model is already cached and the low_cpu profile is requested offline, validation scans the balanced model's files, returns `StartupValidationStatus::Ready`, and `try_initialize` skips the `allow_downloads` gate. The CI workflow reuses a single `MEMPALACE_EMBED_CACHE` for both benchmark runs, making the multi-profile collision non-hypothetical. The reverse failure also applies: a corrupted file in any cached model poisons validation for all profiles.
- Validator file checklist is incomplete â€” partial fastembed caches return false Ready (mempalace-rs/crates/mempalace-embeddings/src/lib.rs): The current checklist requires `onnx`, `tokenizer.json`, and `config.json`. fastembed 5.13.2 also loads `special_tokens_map.json` and `tokenizer_config.json` for text models. A partial download missing either file is reported `Ready`, after which `TextEmbedding::try_new` fails with an opaque backend error rather than a clean startup-validation failure. This is a second, independent false-positive path that the startup validator was introduced to prevent.

## Non-Blocking Issues

- Benchmark publishes cold-start latency as warm baseline; p95 math equals max at 15 samples (mempalace-rs/crates/mempalace-embeddings/examples/embedding_bench.rs): `EmbeddingBenchmark::measure` begins timing on the first post-init request with no warmup iterations. With `MEMPALACE_EMBED_ITERATIONS=15`, `ceil(0.95 Ă— 15) = 15`, so the nearest-rank p95 is always the maximum sample, meaning first-request thread-pool spin-up overhead dominates the published 'warm' number. The CI workflow explicitly labels this as a warm baseline, making the measurement misleading rather than just conservative.
- RSS budget constants disagree with documented Phase 11 targets (mempalace-rs/crates/mempalace-embeddings/src/lib.rs): `ResolvedEmbeddingProfile` hard-codes `low_cpu_idle_rss_budget_mb: Some(700)` and `low_cpu_ingest_rss_budget_mb: Some(1_200)`, while `docs/RustMigrationTasks.md` documents `<= 450 MB` idle and `<= 850 MB` ingest. The fields are unused in Phase 3 but will mislead whoever wires the Phase 11 gate checks. No comment explains the discrepancy.

## Outstanding Concerns

- The validator and the fastembed backend use separate sources of truth: validation is a homegrown recursive file scan, while initialization delegates to fastembed's actual model metadata and HF cache logic. Fixing the immediate bugs by scoping the scan to the correct subdirectory is necessary but may not be sufficient; a future model format change in fastembed will silently invalidate the checklist again without any mechanism for detection.
- The `low_cpu` profile swap from `paraphrase-MiniLM-L3-v2` (architecture reduction, 3 layers) to `AllMiniLML6V2Q` (quantization of a 6-layer model) makes a materially different performance trade-off.
