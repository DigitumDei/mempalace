## Executive Summary

This branch implements Phase 5 of the Rust search and layered-memory runtime in mempalace, delivering embedding-backed search, room/wing filtering, layered recall (L1/L2/L3), and wake-up output generation. Both reviewers agree on four substantive blocking defects: the Lance storage backend silently truncates `list_drawers` to an arbitrary first 10k rows, making wake-up and recall sort a partial dataset on any real corpus; deterministic tie-breaking at the `top_k` cutoff is illusory because equal-distance candidates beyond the requested limit are discarded by the backend before Rust sees them; `generate_layer1` accounts against a character budget using byte-length calls, silently over-running for any non-ASCII memory content; and two HOME-environment tests mutate process-global state under parallel `#[test]` execution, creating an undefined-behavior race. The test suite is entirely stub-backed and cannot surface any of these backend correctness failures, which is the root reason CI passage would not constitute meaningful validation.

## Blocking Issues

- `list_drawers` silently truncates to 10,000 rows (mempalace-rs/crates/mempalace-storage/src/lance.rs): The Lance backend hard-codes `.limit(10_000)` with no ordering guarantee. Wake-up (`generate_layer1`) and recall both call `list_drawers` and sort the returned slice locally. On any corpus larger than 10k drawers the sort operates over an arbitrary partial subset, making ranking and layer assembly incorrect. This directly violates the retrieval-parity and stable-output requirements.
- Deterministic tie-breaking at `top_k` cutoff is not guaranteed (mempalace-rs/crates/mempalace-search/src/lib.rs): `search` passes `limit: query.limit` straight to the storage backend, which returns exactly that many rows. `rank_matches` applies a deterministic secondary comparator only over what the backend chose to return; equal-distance candidates that straddled the cutoff are silently discarded upstream and never reach the Rust layer. The existing tests avoid this because the in-memory stub returns all candidates unconditionally.
- `generate_layer1` uses byte counts against a character budget (mempalace-rs/crates/mempalace-search/src/lib.rs): `room_line.len()` and `entry.len()` return UTF-8 byte lengths. The config field is `max_chars` and the constant is `DEFAULT_LAYER1_MAX_CHARS`. For any non-ASCII memory content (emoji, accented text, CJK) the budget is under-counted, allowing more output than intended. All test fixtures are ASCII so no test exposes this. Fix requires `.chars().count()` at both call sites.
- HOME environment-variable mutation races under parallel test execution (mempalace-rs/crates/mempalace-search/src/lib.rs): `wake_up_request_default_reads_home_identity_when_present` and `wake_up_request_default_falls_back_when_home_identity_is_missing` both call `unsafe { std::env::set_var("HOME", ...) }` without a serialization guard. `cargo test` runs `#[test]` functions in parallel threads by default. Concurrent HOME mutations—or one mutation concurrent with any test reading `default_identity_path()`—is undefined behavior and can produce flaky CI.

## Non-Blocking Issues

- Orphan room header emitted before truncation sentinel (mempalace-rs/crates/mempalace-search/src/lib.rs): `generate_layer1` unconditionally pushes `room_line` before checking whether any entry fits within the remaining budget. When truncation triggers immediately after the header push, the rendered output contains a `[room-name]` header with no entries followed by the `... (more in L3 search)` sentinel.
- `recall(limit=0)` silently defaults to 10 while `search(limit=0)` returns empty (mempalace-rs/crates/mempalace-search/src/lib.rs): The asymmetry between the two public functions for the zero-limit case is undocumented and untested. Whether intentional or not, it should be pinned by a test and noted in documentation.
- Phase plan checklist overstates test fidelity (docs/rust-phase-plans/Phase05-Search-And-Layered-Memory.md): The plan marks golden tests and layered integration tests as complete, but the patch delivers hand-written string assertions against a stub store—not fixture-file golden comparisons or real backend integration tests. Core functionality is present; the test methodology described in the checklist is not.
- Crate-wide `#![allow(missing_docs)]` suppresses workspace lint (mempalace-rs/crates/mempalace-search/src/lib.rs): The workspace `Cargo.toml` sets `missing_docs = "warn"`. The crate-level attribute silently overrides this for the entire public API including `SearchRuntime`, `WakeUpRequest`, and `render_search_results`.

## Missing Tests

- Deterministic ranking test where equal-distance candidates straddle the `top_k` cutoff
- Actual golden-file comparisons for `search-cli.txt`, `search-programmatic.json`, and `wake-up*.txt` instead of hand-written string assertions

## Disputed Issues

- `normalize_score` producing negative values: plausible for L2/unnormalized cosine distances, but the actual distance metric emitted by the real Lance backend is not visible in this diff; one reviewer considers it non-blocking pending backend evidence
- BTreeMap alphabetization of rooms in `generate_layer1` losing importance rank: structurally confirmed, but the phase plan spec asks for stable output rather than importance-ordered room display; whether alphabetical grouping is the intended stable ordering is unresolved
- Redundant distance tiebreaker in `compare_ranked_matches`: one reviewer considers it potentially misleading dead code; the other considers it a harmless determinism aid for float-precision edge cases

## Outstanding Concerns

- The entire test suite is stub-backed and self-fulfilling: the stub store returns data in exactly the shape the search crate expects, meaning all tests confirm internal consistency but provide no evidence of correctness against the real Lance backend. CI passage on this patch would not constitute meaningful integration validation.
- The intent behind the 10k row cap in `list_drawers` is undocumented. If it is a deliberate defensive bound (to limit memory during full-table scans), the real defect is missing documentation and the absence of an error or warning when the cap is hit; if it is an unintentional omission, the fix must include pagination or a backend-side ordering and limit push-down.
- Whether the real Lance backend imposes a consistent secondary ordering (e.g., insertion order) for equal-distance results is unknown. If it does, the tie-at-cutoff problem may produce stable but incorrect output rather than flaky output—still wrong to rely on, but the failure mode differs.

### Claude (claude / claude-sonnet-4-6)

```text
**Blocking Issues**

**1. `generate_layer1` tracks bytes against a character budget.**
`room_line.len()` and `entry.len()` return UTF-8 byte counts. The config field is named `max_chars` and the constant is `DEFAULT_LAYER1_MAX_CHARS`. For ASCII-only content this is harmless, but any Unicode memory content (emoji, accented characters, CJK text) will under-count against the budget and allow more output than intended. The tests use ASCII fixtures so they pass without exposing this. The fix is `room_line.chars().count()` and `entry.chars().count()`, and likely a rename of `total_len` to make the intent explicit.

**2. Environment-variable race condition in tests.**
`wake_up_request_default_reads_home_identity_when_present` and `wake_up_request_default_falls_back_when_home_identity_is_missing` both use `set_home`/`restore_home` around `unsafe { std::env::set_var("HOME", ...) }` without any test-serialization guard. `cargo test` runs `#[test]` functions in parallel threads within the same binary by default. Two concurrent HOME mutations—or one mutation concurrent with any other test reading `default_identity_path()`—is undefined behavior and will produce flaky CI. These need either a per-process mutex (`std::sync::Mutex` in a `LazyLock`) or `#[serial_test::serial]`.

---

**Non-Blocking Issues**

**1. `normalize_score` is unclamped.**
`1.0 - distance` produces negative scores if the storage backend returns distances greater than 1.0 (which is valid for L2 or unnormalized cosine distances). The rendered output (`Match:  -0.267`) would be confusing. There is no documented assumption that distances are constrained to `[0, 1]`, and the stub store in tests uses raw f32 differences (`|embedding[0] - query[0]|`) which can be arbitrarily large.

**2. `generate_layer1` room ordering loses importance rank.**
`order_layer_drawers` sorts drawers by `layer_weight` descending, which determines which drawers make the `top` cut. But they are then inserted into a `BTreeMap<String, Vec<DrawerRecord>>` which reorders rooms alphabetically. A high-importance room named `zygote` would always appear after a low-importance room named `alpha`. The sort before grouping only controls record selection and per-room record ordering, not room output order. The test passes because `auth-migration < backend < phase0-rollout` is both alphabetical and importance order in the fixture—a coincidence, not a guarantee.

**3. Orphan room header on truncation.**
When `total_len + entry.len() > max_chars`, `room_line` has already been unconditionally pushed to `lines`. The truncation sentinel `... (more in L3 search)` is appended after it. Rendered output will contain a `[room-name]` header with no records beneath it before the truncation line. This is cosmetically wrong.

**4. Redundant distance tiebreaker in `compare_ranked_matches`.**
Score is `1 - distance`, so sorting by score descending is identical to sorting by distance ascending. The `.then_with(|| compare_distance(left.distance, right.distance))` branch can only matter if f32 rounding causes two items with distinct distances to round to the same score, which is possible but rarely intentional. No comment explains why this tiebreaker exists.

**5. `recall` with `limit=0` silently defaults to 10.**
`search` returns an empty vec for `limit=0`. `recall` converts it to 10. This asymmetry is undocumented and untested.

**6. Crate-wide `#[allow(missing_docs)]`.**
Suppresses documentation lints for the entire public API, including `SearchRuntime`, `WakeUpRequest`, `IdentitySource`, and `render_search_results`. If the workspace enforces documentation elsewhere this is a regression.

---

**Missing Tests**

- Room-only filter in `search` (wing=None, room=Some). Existing test always pairs wing+room together for the combined-filter case.
- `recall` with `limit=0` to pin the default-to-10 behavior and distinguish it from `search`'s empty-result behavior.
- `generate_layer1` with an empty store (the `## L1 — No memories yet.` branch is only exercised via `wake_up`, not directly).
- `search_text` with actual results: only the empty-results path is tested.
- `DrawerMatch` with `distance=None` through `rank_matches`: `normalize_score(None)` returns `0.0` but `compare_distance(None, None)` returns `Equal`—the interaction is untested.
- `wake_up` with an empty store.
- Non-ASCII content in `generate_layer1` to exercise (and expose) the byte/char budget mismatch.
- `trim_similarity` unit tests for edge values (0.0, 1.0, negative, >1.0): the current tests only cover values that happen to work via the render integration tests.

---

**Strong Concerns**

The byte/char confusion in `generate_layer1` is the most likely to cause a real production regression: memory content is often non-ASCII (project names, personal notes, emoji tags), and the budget will silently overrun without any visible error. The test suite is incapable of catching it because all fixtures are ASCII.

The parallel `HOME` mutation is a correctness issue hiding behind luck. If the two HOME tests are scheduled to opposite ends of the test run they appear fine; on a machine with high thread contention or under `--test-threads=1` they will interfere. This should be caught before merge.

---

**Confidence**

High on the byte/char and HOME-race issues—both are clearly present in the diff. Moderate on the score-clamping concern (depends on what distances the real storage layer emits, which is not visible in this diff). The room-ordering observation is structural and reproducible from the BTreeMap insertion alone.
```
