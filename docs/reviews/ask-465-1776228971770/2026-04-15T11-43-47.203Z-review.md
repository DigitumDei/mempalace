## Executive Summary

This branch implements the Phase 5 Rust search and layered-memory runtime for mempalace, replacing placeholder behavior with real embedding-backed search, wing/room filtering, ranked results, and layered-memory assembly for recall and wake-up flows. Two unambiguous blocking defects emerge from cross-reviewer consensus: `search()` loads the entire filtered corpus into memory just to obtain a count, then passes that count as the vector search limit, making every search a full-corpus materialization that defeats the LanceDB ANN index; and `generate_layer1()` silently drops already-accepted entries when character-budget truncation fires mid-room, because the `room_lines` buffer is never flushed before the early return. Both defects are directly traceable through the control flow and are not caught by existing tests. The verdict is revise because neither blocking issue has been addressed and together they break the branch's stated goals of scale-safety and correct output construction.

## Blocking Issues

- `search()` full-corpus materialization via `list_drawers(...).len()` (mempalace-rs/crates/mempalace-search/src/lib.rs): All four reviewers independently confirmed this defect across two rounds. `SearchRuntime::search` calls `store.list_drawers(&filter).await?.len()` (lib.rs:163) to compute `candidate_count`, loading every matching `DrawerRecord` — including full content and embedding vectors — into memory solely to read its length, then immediately passes `candidate_count` as the `limit` to `search_drawers` (lib.rs:168). On the Lance backend, `list_drawers` performs a `count_rows` call followed by a full `query().limit(row_count)` with complete deserialization (lance.rs:256–270), meaning the entire filtered corpus is materialized twice per search. The `filtered_row_count` helper added to `LanceDrawerStore` is on the concrete type, not the `DrawerStore` trait, and is unreachable from `SearchRuntime`. This directly contradicts the branch's stated goal of being safe for larger datasets.
- `generate_layer1()` mid-room truncation silently drops committed entries (mempalace-rs/crates/mempalace-search/src/lib.rs): Confirmed by all reviewers in round 2. Accepted entries for the current room are buffered in `room_lines` (lib.rs:378); `lines.extend(room_lines)` only executes at the end of the per-room inner loop (lib.rs:381). When the character-budget check at line 366 fires on the second or later entry of a room, the room header is already in `lines` (pushed at line 372 when `room_has_entries` first became true) but the early return at line 368 skips the `extend`, discarding all previously accepted entries from that room. The output contains an orphan room header immediately followed by `... (more in L3 search)` with no entries. The existing test `generate_layer1_does_not_emit_orphan_room_headers_on_truncation` does not cover this case — it sets a budget that rejects the first entry of room 'beta' before `room_has_entries` is set, so no entries are buffered when the return fires.

## Non-Blocking Issues

- `recall()` and `generate_layer1()` load the full filtered corpus in memory before sorting and truncating (mempalace-rs/crates/mempalace-search/src/lib.rs): Both functions call `list_drawers` to fetch every matching `DrawerRecord`, sort in-process, then keep only 10 or 15 results. This is the same shape as the `search()` blocking defect but is classified non-blocking because the current `DrawerStore` trait lacks a server-side ordered top-N primitive; a clean fix requires trait changes likely outside Phase 5 scope.
- `default_identity_path()` falls back to a literal `~` path when HOME is unset (mempalace-rs/crates/mempalace-search/src/lib.rs): Confirmed by Gemini and Codex in round 2. `PathBuf::from("~")` is not shell-expanded by Rust's `std::fs`; when HOME is unset the code looks for a directory literally named `~` in the CWD. Practical impact is soft: the `DefaultPath` match arm handles `NotFound` by returning the banner rather than propagating an error, so the failure mode is silent fallback rather than a crash.
- `recall(limit=0)` silently defaults to 10; `search(limit=0)` returns empty (mempalace-rs/crates/mempalace-search/src/lib.rs): Confirmed by Claude across both rounds. `recall()` line 201 substitutes 10 for a zero limit; `search()` line 147 returns `Vec::new()` immediately. `LayerRetrieveRequest::default()` has `limit: 0`, meaning callers using the default receive 10 results with no indication the value was substituted. The behaviors differ and neither is documented or tested.
- `generate_layer1` character budget does not count `join`-injected newline separators (mempalace-rs/crates/mempalace-search/src/lib.rs): Confirmed by Claude in both rounds; not disputed. `total_chars` tracks individual string lengths but `lines.join("\n")` adds one `\n` per non-final line, none of which are counted. `max_chars` therefore functions as a lower bound rather than a hard cap. The Python Layer 1 budget logic has the same omission, so this is not a Rust-specific regression, but it means the configured limit is softer than documented.
- `order_layer_drawers` sorts by room name before recency on equal-weight ties (mempalace-rs/crates/mempalace-search/src/lib.rs): Confirmed as a real sort-order observation by all reviewers (lib.rs:426–438): weight desc → room asc → date desc → filed_at desc. On equal weight, an older drawer in room 'alpha' beats a newer drawer in room 'zeta'. Severity is disputed (see disputedIssues): the Python docstring says 'highest-weight / most-recent' but the Python implementation at layers.py:131 sorts only on importance with no recency tie-break, making the normative contract ambiguous. Classified non-blocking absent a clear authoritative spec violation.

## Disputed Issues

- `trim_similarity(0.49)` f32 formatting bug: Claude argues concretely that `0.49_f32 * 100.0` leaves a residual of ~9.5e-7 exceeding `f32::EPSILON`, causing the `.3` branch to fire and emitting `"0.490"` instead of `"0.49"`, masked by a loose `contains("Match:  0.49")` assertion; Codex round 2 asserts this does not reproduce and marks the claim invalid. Neither produced a passing/failing test run to settle it.
- `normalize_score` producing negative values: Claude flags that cosine distances above 1.0 yield negative scores with no clamping; Codex notes Python also uses `1 - dist` and that the claim needs a concrete failing case against the real LanceDB metric rather than a hypothetical. Not confirmed or refuted by code inspection alone.
- `order_layer_drawers` blocking vs non-blocking severity: Gemini round 2 classifies the room-before-recency sort as a third blocking issue; Claude and the round 2 judge hold it non-blocking because the Python reference implementation does not enforce recency tie-breaking either, making the docstring contract ambiguous rather than violated.

## Outstanding Concerns

- The `DrawerStore` trait lacks a `count_drawers` or server-side ordered-top-N primitive. Until the trait is extended, `recall()` and `generate_layer1()` cannot avoid full-corpus materialization, and any fix for `search()` must use a bounded overfetch heuristic rather than a clean count query. This is a trait design gap that will affect every future backend implementation.
- The branch pursues strict mathematical determinism in ranking by overfetching the entire corpus and re-sorting locally. The round 1 judge noted this is the wrong architectural tradeoff for the stated scale-safety goal. A bounded-overfetch strategy (fetch query.limit plus a small constant margin, sort locally) would preserve practical determinism without O(corpus) reads.
- Retrieval golden tests, wake-up golden tests, layered output integration tests, and tolerant quality checks are explicitly deferred per the Phase 5 plan checklist. There is consequently no regression guard on full output shape or end-to-end ranking quality — acceptable as planned scope residual, but noted as a risk surface.

### Claude (claude / claude-sonnet-4-6)

```text
**Blocking Issues**

**1. `search()` calls `list_drawers` solely to count candidates, discards all records.**

```rust
let candidate_count = store.list_drawers(&filter).await?.len();
```

This fetches every matching `DrawerRecord` (including full content and embedding vectors) just to get the count, immediately discards the data, then does a separate `search_drawers` call. For a 10,000-drawer wing this allocates and drops ~10k full records on every search. The `DrawerStore` trait has no `count` method; the `filtered_row_count` helper added to `LanceDrawerStore` is private and unreachable from the runtime. The right fix is either a trait-level count method or accepting that `search_drawers` with a very large limit is the overfetch primitive, not a separate list call.

**2. `normalize_score` is unbounded and produces negative values.**

```rust
fn normalize_score(distance: Option<f32>) -> f32 {
    distance.map_or(0.0, |value| 1.0 - value)
}
```

LanceDB's cosine distance for unnormalized vectors can exceed 1.0, producing negative scores. The rendered output would show e.g. `Match: -0.267` with no indication it's invalid. No clamping exists anywhere in the pipeline. There is no test for `distance > 1.0`. This is not hypothetical: the test fixture uses importance scores (`Some(-0.267)`, `Some(-0.848)`) which, when used as the stub embedding's first component, produce `abs()` distances that are ≤ 1.0 in the stub but do not reflect what real cosine distance can return.

**3. `trim_similarity` epsilon comparisons are unreliable for f32.**

```rust
} else if ((rounded * 100.0).round() - (rounded * 100.0)).abs() < f32::EPSILON {
    format!("{rounded:.2}")
```

`f32::EPSILON` is ~1.19e-7. After `f32 * 100.0`, floating-point rounding residuals are typically ~1e-7 to 1e-6, which may or may not satisfy the epsilon test depending on the exact f32 representation of the input. For `0.49_f32 * 100.0`, the residual is borderline. The tests use substring matching (`contains("Match:  0.49")`) which passes whether the output is `"0.490"` or `"0.49"`, so the tests do not actually validate the format. A value that should format as 2 decimal places may silently emit 3.

---

**Non-Blocking Issues**

**4. `list_drawers` in Lance now makes two sequential DB round trips.**

The fix replaces `limit(10_000)` with `count_rows` + `query().limit(row_count)`. This is correct for correctness but introduces a TOCTOU window (rows inserted between the two calls are missed) and doubles DB latency for every `list_drawers` call. For the Lance backend this is a regression in performance for the common case. Worth documenting at minimum.

**5. `recall()` limit=0 silently becomes 10; `search()` limit=0 returns empty.**

```rust
let limit = if request.limit == 0 { 10 } else { request.limit };
```

`LayerRetrieveRequest::default()` has `limit: 0`, which maps to 10 here. In `search()`, `limit == 0` returns `Vec::new()` immediately. The semantics differ and are undocumented. There is no test for `recall` with `limit: 0`.

**6. `generate_layer1` char budget ignores `\n` join separators.**

The budget tracks character counts from individual strings but the final output is `lines.join("\n")`. The `\n` characters between entries are not counted. Room headers start with `"\n[room]"` (one explicit newline), but entry lines and the L1 header line have join-injected newlines not tracked. The Unicode char budget test passes only because it's designed around the implementation's own counting convention, not the actual output byte count.

**7. `source_label()` allocates in every sort comparison.**

`compare_ranked_matches` calls `source_label()` twice per invocation, allocating two new `String`s per comparison step in an O(n log n) sort. Precomputing labels once before sorting would eliminate this.

**8. `unsafe` env mutation in tests is a soundness risk.**

`HomeEnvGuard::set` uses `unsafe { std::env::set_var(...) }` under a mutex. The mutex serializes tests within this module, but other crates or test modules running concurrently in the same binary that read `HOME` without holding the mutex would see intermittent values. Rust 2024 makes `set_var` formally unsafe for this reason. This is a standard workaround, but concurrent test runners can still trigger the race.

---

**Missing Tests**

- `trim_similarity` with exact round-number inputs (0.5, 0.25, 0.1, 0.125) to verify decimal place selection is stable across f32 representations.

- `recall()` with `limit: 0` — verifies the silent default-to-10 behavior is intentional.
- `search()` with a store containing >10,000 drawers — the Lance regression test exists but no equivalent in the search runtime.
- Embedding profile mismatch between stored data and query (currently only provider-vs-query mismatch is tested, not stored-vs-query).
- Retrieval golden tests, wake-up golden tests, layered output integration tests, tolerant quality checks — all explicitly deferred in the plan.

**Strong Concerns**

**Architecture: search overfetch via `list_drawers`.**
The `DrawerStore` trait is missing a primitive that `search()` clearly needs: a filtered count. Without it, the runtime is forced to load all candidate records twice per search (once for count, once for vector search results). This is a trait design gap that will affect every non-Lance backend implementing `DrawerStore`. The plan should either add a `count_drawers` method to the trait or replace the two-phase pattern with a single overfetch of `usize::MAX` to `search_drawers` with re-ranking applied entirely in-process.

The overfetch architecture is a performance and design problem that will get worse with scale. The explicitly deferred golden tests mean there is no regression guard on full output shape.
