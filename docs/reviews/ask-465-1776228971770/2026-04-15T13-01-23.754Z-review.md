## Executive Summary

This branch delivers Phase 5 search and layered-memory functionality for the Rust port, covering embedding-based retrieval, wing/room filtering, formatted output, L1/L2 recall assembly, and substantial test coverage. Two blocking issues were confirmed by panel consensus: deterministic top-k ranking is not structurally guaranteed because the backend overfetch margin (limit+32) means tie groups larger than 32 around the cutoff are arbitrarily truncated before Rust's lexicographic sort runs, and Layer 1 character-budget accounting diverges from the Python reference by ~24 characters because Rust counts the L1 header against max_chars while Python does not. These defects directly contradict the stated success criteria of deterministic ranking and CLI/Python-parity output, making the branch unready for PR without fixes.

## Blocking Issues

- Deterministic top-k not guaranteed for tie groups larger than overfetch margin (mempalace-rs/crates/mempalace-search/src/lib.rs): search() fetches limit+32 candidates from Lance, which truncates to that count before Rust's lexicographic tie-break sort runs. If more than 32 records share the same score at the cutoff, the backend returns an arbitrary subset, permanently excluding lexicographically earlier winners from the final top-k. The existing tests use only 2-drawer stores and cannot detect this failure. The stated success criterion requires deterministic, correctly ordered top-k results with stable tie-breaking.
- Layer 1 character budget counting diverges from Python: Rust counts the L1 header, Python does not (mempalace-rs/crates/mempalace-search/src/lib.rs): Rust seeds total_chars with the length of the '## L1 — ESSENTIAL STORY' header (~24 chars) before the room/entry accumulation loop. Python initializes total_len = 0 and never counts the header against MAX_CHARS. For the same configured max_chars, Rust truncates approximately 24 characters earlier than the Python reference, a direct parity regression in wake-up/L1 output. Unit tests are calibrated to the Rust undercounting and therefore cannot detect this gap.

## Non-Blocking Issues

- recall(limit=0) silently substitutes 10; search(limit=0) returns empty (mempalace-rs/crates/mempalace-search/src/lib.rs): The asymmetric behavior is undocumented and untested. A caller passing limit=0 to LayerRetrieveRequest expecting zero-or-empty semantics receives 10 results instead.
- HOME fallback produces a literal '~' path rather than expanding to the user's home directory (mempalace-rs/crates/mempalace-search/src/lib.rs): default_identity_path() falls back to PathBuf::from('~') when HOME is unset. Rust's filesystem APIs do not expand tildes, so this is treated as a literal directory name and silently falls back to the banner. Python uses os.path.expanduser which queries pwd when HOME is absent, making the Rust behavior weaker and environment-dependent.
- list_drawers materializes full DrawerRecord including embedding vectors before Rust-side truncation (mempalace-rs/crates/mempalace-storage/src/lance.rs): generate_layer1 and recall both call list_drawers, which rebuilds complete DrawerRecord structs including embedding vectors, then take(max_drawers) discards most of them in Rust. For a Balanced profile with ~1536 f32 floats per embedding and a 10k-drawer store, this loads approximately 60 MB before any truncation occurs.
- source_label() allocates a String inside sort comparators at O(N log N) frequency (mempalace-rs/crates/mempalace-search/src/lib.rs): source_label() returns an owned String and is called inside the sort_by closures in both compare_ranked_matches and order_layer_drawers, causing heap allocations on every comparison during sort. Precomputing the label or returning a borrowed type would eliminate this.
- generate_layer1 total_chars undercounts actual output length by the number of join separators (mempalace-rs/crates/mempalace-search/src/lib.rs): total_chars tracks the lengths of individual lines but ignores the \n bytes inserted by lines.join('\n'). For N elements, the actual rendered output is N-1 bytes longer than total_chars, meaning the budget check allows slightly more output than max_chars. Tests replicate the same undercount so they pass but validate a consistently wrong boundary.

**Blocking Issues**

**`trim_similarity` is functionally broken.** The `f32::EPSILON` (≈1.19e-7) threshold used to detect "is this value representable with N decimal places" is meant for ULP-level differences, not for "is the 10× multiple of this value close to an integer." For a score like 0.49f32 (which is approximately 0.49000000953674316 in IEEE 754), the level-1 check `|(0.49 * 10.0).round() - (0.49 * 10.0)|` ≈ 0.1 — eleven orders of magnitude above `f32::EPSILON`. The level-2 check `|(0.49 * 100.0).round() - (0.49 * 100.0)|` ≈ 9.5e-7, still 8× above `f32::EPSILON`. So 0.49 falls through to the `.3` branch and formats as "0.490". Tests do not catch this because they use `.contains("0.49")`, and "0.490" contains "0.49" as a substring. The consequence is that every score value that isn't exactly representable at 1 or 2 decimal places in f32 gets a spurious trailing zero.
