## Executive Summary

This branch implements the Phase 5 Rust search runtime for mempalace, covering embedding-based query search, room/wing filter propagation, normalized scoring, recall assembly, and wake-up/L1 generation against canonical raw drawers only. Both reviewers independently identified two deterministic test assertion failures — a score value mismatch caused by treating distance as the final score rather than applying `normalize_score`, and a recall header format string mismatch — that will cause CI to reject the branch as-is. A third consensus finding is that `WakeUpRequest::default()` inverts the Python reference contract by never attempting to read `~/.mempalace/identity.txt`, creating a parity gap for any caller relying on the default. The verdict is revise because the two test failures are deterministic and directly reproducible from the code without running a build.

## Blocking Issues

- Score assertion wrong in `search_applies_filters_and_normalizes_similarity` (mempalace-rs/crates/mempalace-search/src/lib.rs): The stub store computes `distance = |0.49 - 0.0| = 0.49`. `rank_matches` then calls `normalize_score` which returns `1.0 - 0.49 = 0.51`. The test asserts the score is `0.49`, not `0.51`. This is a deterministic failure — the test was written as if `normalize_score` is the identity function. Both reviewers confirmed the code path independently.
- Recall header format mismatch in `recall_returns_stable_filtered_layer_output` (mempalace-rs/crates/mempalace-search/src/lib.rs): The format string at the implementation site expands to `"## L2 — ON-DEMAND (2) drawers"` (count in parens, word outside), but the test asserts `starts_with("## L2 — ON-DEMAND (2 drawers)")` (both inside parens). `starts_with` returns false and the assertion fails. Both reviewers confirmed the mismatch from the format string.

## Non-Blocking Issues

- `WakeUpRequest::default()` uses `MissingDefault` instead of attempting the identity file (mempalace-rs/crates/mempalace-search/src/lib.rs): The Rust `Default` impl sets `identity: IdentitySource::MissingDefault`, which emits only the fallback banner. The Python reference defaults to reading `~/.mempalace/identity.txt` and falls back only when absent. No existing test exercises this path via `default()`, so it will not fail CI, but it is a real API contract divergence from the Python behavior.
- `drawer_id: None` hardcoded in search result mapping (mempalace-rs/crates/mempalace-search/src/lib.rs): The hit-mapping step unconditionally sets `drawer_id: None` even though `DrawerRecord.id` carries a real value. `SearchResult` explicitly supports carrying the ID. This silently drops traceability for downstream callers. Whether the field is intentionally left empty in this phase is undocumented.
- Em dash vs hyphen-minus in fallback identity banner (mempalace-rs/crates/mempalace-search/src/lib.rs): Rust emits `## L0 - IDENTITY` (U+002D hyphen-minus); Python emits `## L0 — IDENTITY` (U+2014 em dash). Minor but a concrete parity divergence for any consumer comparing output strings.

### Claude (claude / claude-sonnet-4-6)

```text
**Blocking Issues**

**1. Score assertion wrong in `search_applies_filters_and_normalizes_similarity` (lib.rs:~750)**

The test asserts `(results[0].score - 0.49).abs() < 1e-6`. The record's importance is `Some(0.49)`, which is also used as its embedding value via `embedding(score.unwrap_or(0.0))`. The stub store computes distance as `(0.49 - 0.0).abs() = 0.49`. Then `normalize_score(Some(0.49)) = 1.0 - 0.49 = 0.51`. The assertion checks for 0.49 but the actual score is 0.51. This test will fail CI with an assertion error. The bug is in the test, not the normalization logic — the test was written as if `normalize_score` is the identity function.

**2. Recall header format mismatch in `recall_returns_stable_filtered_layer_output` (lib.rs:~815)**

The implementation produces `"## L2 — ON-DEMAND (2) drawers"` (number inside parens, word outside). The test asserts `rendered.starts_with("## L2 — ON-DEMAND (2 drawers)")` (number and word both inside parens). These strings differ and `starts_with` will return false. The format string at line ~250 is `format!("## L2 — ON-DEMAND ({}) drawers", ...)` — the `{}` expands to `(2)` followed by ` drawers`, not `(2 drawers)`.

Both are deterministic compile-and-run failures. The checklist marks all test items as `[x]` done, but CI will reject this branch.

---

**Non-Blocking Issues**

**3. `trim_similarity` float-comparison logic is broken for common values**

The epsilon checks use `f32::EPSILON` (≈1.19e-7) to decide whether a value is representable with fewer decimal places, but after multiplying by 10 or 100, accumulated float error typically exceeds that bound. For score 0.49: `0.49 * 100.0` in f32 ≈ 48.9999..., `(49.0 - 48.9999...).abs() ≈ 2e-6 >> f32::EPSILON`, so the function falls through to `.3` format and emits `"0.490"` instead of `"0.49"`. The tests still pass because they use `contains("Match:  0.49")` and `"0.490"` does contain `"0.49"` as a substring, masking the defect.

**4. `normalize_score` has no clamping**

`1.0 - distance` is unclamped. If the storage backend returns distances outside `[0, 1]` (Euclidean, dot-product, or cosine on unnormalized vectors can all do this), scores go below 0 or above 1 with no error or truncation. There is no contract enforcement on the distance range.

**5. Redundant secondary sort by distance in `compare_ranked_matches`**

`score = 1.0 - distance`, so equal scores imply equal distances (for non-None cases). The `compare_distance` tiebreak at the second level is a no-op for all normal cases and only activates for two results that both have `distance = None` (Equal) or a floating-point coincidence. The code works correctly but the dead branch is misleading.

**6. `order_layer_drawers` result is overwritten by BTreeMap in `generate_layer1`**

`order_layer_drawers` sorts by `layer_weight` descending to select the top-k records, but then `generate_layer1` groups them into a `BTreeMap<String, Vec<DrawerRecord>>` which re-orders rooms alphabetically. The final presentation order is entirely alphabetical by room name, not layer-weight ordered. The sort only affects selection, not presentation. This is not a bug in the current code, but it is a subtle indirection that will mislead future maintainers who expect the sort to influence output order.

---

**Missing Tests**

- **`generate_layer1` truncation path**: the `"... (more in L3 search)"` branch triggered when `total_len + entry.len() > config.max_chars` is never exercised. The `wake_up` test sets `max_chars: 3_200` which is large enough to include all sample data without truncation.
- **Blank query → `BlankQuery` error**: guard exists at the top of `search()`, no test.
- **`limit = 0` → empty vec early return**: guard exists, no test.
- **`recall` with empty store**: `search_text_reports_empty_results` covers the search empty-result path but there is no equivalent test for `recall` returning `"No drawers found."`.
- **`IdentitySource::Path`**: only `Inline` and `MissingDefault` are tested; the file-read path and its `IdentityRead` error are untested.
- **Combined wing + room filter in `search`**: only wing-only filter is tested.
- **`trim_similarity` directly**: no unit tests for the formatting function on clean values like `0.5`, `1.0`, or `0.0`.

---

**Strong Concerns**

The two blocking bugs share a common cause: the tests appear to have been written from an incorrect mental model of the computation (treating distance as the final score, not `1.0 - distance`) and from a format string that was misread. Both are pure test-authoring errors, not subtle logic issues, which makes it more concerning that the checklist flags these as complete and validated. High on the `trim_similarity` float-epsilon analysis. Medium on the missing-test list — the codebase may have integration tests or golden-test infrastructure elsewhere that cover some of these, but nothing in this diff covers them.
```
