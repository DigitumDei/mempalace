## Executive Summary

This branch implements Phase 11 low-CPU hardening by wiring configurable resource limits — thread caps, queue bounds, ingest batch sizing, search result caps, degraded mode, and reranking control — across the CLI, search, and MCP paths. Two consensus blocking issues prevent merge: the rerank ordering test is deterministically wrong because both fixtures fully match all three query terms (identical lexical scores of 1.0), leaving the lower-distance record as the winner and making the assertion on backend-2.md provably fail CI; and the write-path `find_duplicates` function now silently examines only 3 instead of 5 candidates under the default degraded low-CPU profile, meaning near-duplicates ranked 4th or 5th are invisibly missed and written as new records without error. The verdict is revise.

## Blocking Issues

- Rerank test asserts the wrong winner and will deterministically fail CI (mempalace-rs/crates/mempalace-search/src/lib.rs): Both fixture records in `rerank_policy_requests_extra_candidates_and_prefers_lexical_overlap` contain all three query terms ('session', 'refresh', 'auth'), so both receive a lexical score of 1.0. With normalize_score(d) = 1.0 - d and descending composite sort, backend.md (distance 0.02) scores 0.986 and backend-2.md (distance 0.04) scores 0.972. The test asserts backend-2.md wins, which is wrong. The test also cannot satisfy its stated goal — with identical lexical scores the reranker can never override the distance ranking, so the fixture does not demonstrate the feature it names.
- find_duplicates silently misses near-duplicates under default degraded low-CPU profile (mempalace-rs/crates/mempalace-mcp/src/lib.rs): `find_duplicates` on the write path now uses `effective_search_results_limit().min(5)` to bound its search window. In the default low-CPU degraded profile, `effective_search_results_limit()` returns 3, so `3.min(5) = 3`. Any near-duplicate that ranks 4th or 5th by embedding similarity is now invisible to deduplication — the record is written as a new entry with no error, warning, or log. This is a silent correctness regression on the write path that affects every user of the default low-CPU profile.

## Non-Blocking Issues

- build_runtime duplicated verbatim between CLI and MCP binaries (mempalace-rs/crates/mempalace-mcp/src/main.rs): The same function body appears in both mempalace-cli/src/main.rs and mempalace-mcp/src/main.rs. Any future change to low-CPU threading logic must be made in two places.
- MCP main loads config twice on the non-stub path (mempalace-rs/crates/mempalace-mcp/src/main.rs): Config is loaded once for `build_runtime` and again inside `McpServer::from_default_config(None)`. If the config file changes between the two reads, the Tokio thread ceiling and the server runtime configuration silently diverge.
- wake_up_layer1_config constructs Layer1Config::default() twice (mempalace-rs/crates/mempalace-cli/src/main.rs): One call reads `.max_drawers` and a second call appears in the struct spread `..Layer1Config::default()`. A single construction is sufficient.
- positive_override silently treats zero as 'use default' with no user feedback (mempalace-rs/crates/mempalace-config/src/config.rs): A user writing `queue_limit: 0` in config gets the default value applied with no validation error, log, or documentation that zero is a sentinel. The behavior is surprising and untested.
- candidate_limit .max(query.limit) is dead code (mempalace-rs/crates/mempalace-search/src/lib.rs): `query.limit.saturating_mul(2).max(query.limit)` — for any usize, x*2 >= x always holds (or saturating_mul caps at usize::MAX, which is also >= x). The `.max(query.limit)` branch is never taken.
- Queue semaphore backpressure is unreachable in the production stdio transport (mempalace-rs/crates/mempalace-mcp/src/lib.rs): `serve_transport` processes requests strictly sequentially (read line → handle → write → read next), so there is never more than one in-flight request in the normal MCP stdio path. The `try_acquire_owned` rejection path is only exercisable by manually spawning concurrent tasks as the test does, not through the real transport.
- Non-ASCII tokenization degrades reranking quality for non-English content (mempalace-rs/crates/mempalace-search/src/lib.rs): `normalized_terms` splits on `!c.is_ascii_alphanumeric()`, which discards all non-ASCII characters and treats them as delimiters. For CJK, Arabic, Cyrillic, or accented Latin content, every token is stripped, producing empty term arrays and a 0.0 lexical overlap score — effectively disabling reranking for non-English content.
- Low-CPU override block silently ignored for Balanced profile (mempalace-rs/crates/mempalace-config/src/config.rs): `with_overrides` stores user-specified values even when `enabled = false`, but all `effective_*` methods short-circuit for Balanced. A user writing `low_cpu: { queue_limit: 10 }` in a Balanced config receives no warning that the block is inert.
- Reranked composite scores used as duplicate-detection threshold when rerank_enabled=true (mempalace-rs/crates/mempalace-mcp/src/lib.rs): When reranking is enabled, `SearchResult.score` becomes a blended 0.7*base + 0.3*lexical value. The `result.score >= threshold` filter in `find_duplicates` was calibrated against pure similarity values. A semantically near-duplicate record with different terminology may score below the threshold on the composite scale and be written as a new entry. This is a structural semantic mismatch on the write path, though it requires `rerank_enabled: true` (non-default in low-CPU profile).

## Outstanding Concerns

- The queue/backpressure semaphore provides admission control in theory but is structurally unreachable in the production stdio transport because requests are processed one at a time. The mechanism is tested in isolation but not in any scenario that reflects real server behavior.
- Non-ASCII content silently receives 0.0 lexical overlap scores due to ASCII-only tokenization, making reranking a no-op for the entire class of non-English queries and documents without any warning or degradation signal.
