## Executive Summary

This branch implements Phase 11 low-CPU hardening for the Rust CLI path by adding a configurable `LowCpuRuntimeConfig`, a custom Tokio runtime builder, ingest batching, and search/wake-up result caps. The core work is real and the CLI does apply `worker_threads`, `max_blocking_threads`, `ingest_batch_size`, `search_results_limit`, and `wake_up_drawers_limit` at runtime. However, three of the seven new config knobs (`queue_limit`, `degraded_mode`, `rerank_enabled`) are parsed and documented but completely inert — no code path reads them after config load — which directly contradicts the stated success criterion of 'supporting degraded-mode limits' and 'enforcing bounded work.' Additionally, the MCP server ignores all low-CPU enforcement: `tool_search` forwards the caller-supplied `limit` without clamping, and the MCP binary never invokes `build_runtime` or the clamping helpers. The verdict is revise because the branch ships advertised, configurable controls that do nothing and leaves the primary daemon transport unhardened.

## Blocking Issues

- queue_limit, degraded_mode, and rerank_enabled are dead config (mempalace-rs/crates/mempalace-config/src/config.rs): All three fields are defined in config.rs, loaded into LowCpuRuntimeConfig, and documented with constants, but no code outside config.rs ever reads them. The runtime builder, CLI commands, and MCP paths are all unaffected by these settings. A user who sets degraded_mode: true or queue_limit: 32 gets no behavioral change, directly contradicting the phase intent of 'supporting degraded-mode limits.'
- MCP search path bypasses low-CPU enforcement (mempalace-rs/crates/mempalace-mcp/src/lib.rs): The CLI enforces search_results_limit and wake_up_drawers_limit through clamping helpers, but mempalace-mcp/src/lib.rs tool_search passes the caller-supplied limit straight to SearchQuery with no cap. The MCP binary also never calls build_runtime, so the constrained Tokio runtime is CLI-only. A client on the MCP transport running on constrained hardware receives none of the hardening this phase was designed to deliver.

## Non-Blocking Issues

- Low-CPU enforcement lives at the CLI boundary rather than in core libraries (mempalace-rs/crates/mempalace-cli/src/main.rs): Limits such as ingest_batch_size and search_results_limit are enforced by the CLI caller rather than inside mempalace-search or mempalace-ingest. This design means any new consumer (MCP, future binaries) must independently re-implement the enforcement. Pushing the policy into core request builders would prevent the MCP gap from recurring.
- clamp_search_results and wake_up_layer1_config lack an explicit low_cpu.enabled guard (mempalace-rs/crates/mempalace-cli/src/main.rs): Both functions rely on usize::MAX as a no-op sentinel for the balanced profile rather than gating on low_cpu.enabled. This works today because balanced defaults resolve to the same values as the library defaults, but the sentinel pattern is fragile: any future path that calls these functions must know about and correctly thread the sentinel, rather than reading an explicit enabled flag.
- positive_override silently discards zero without logging (mempalace-rs/crates/mempalace-cli/src/main.rs): If a user mistakenly sets worker_threads: 0, the value is quietly replaced with the default. There is no warning or error. This will produce hard-to-debug behavior where an explicitly configured value has no effect.
- MCP ingest path does not use low-CPU batch sizing (mempalace-rs/crates/mempalace-mcp/src/lib.rs): The CLI ingest path respects max_embed_batch_size from low-CPU config, but the MCP write path builds embeddings one record at a time without consulting LowCpuRuntimeConfig. The batching hardening is narrower than the change description implies.

## Missing Tests

- No test verifies that the MCP tool_search path respects search_results_limit under low-CPU config — such a test would fail today.
- No test verifies that wake_up_drawers_limit actually limits the drawers returned by the wake-up command path; the layer1 config wiring has no coverage.
- The existing low_cpu_search_results_are_clamped_by_runtime_config test asserts count == 1 but does not verify the unclamped path returns more than one result, so it cannot confirm the clamp is doing work rather than the query coincidentally returning one result.
- No tests cover queue_limit, degraded_mode, or rerank_enabled behavior — consistent with these fields being inert at runtime.
- No test asserts that build_runtime actually applies worker_threads(1) when low_cpu.enabled is true.

## Disputed Issues

- Whether DEFAULT_LOW_CPU_MAX_BLOCKING_THREADS = 1 is a confirmed deadlock risk or merely a plausible concern: Gemini flagged it as a strong concern; Claude and Codex noted it is only a real risk if spawn_blocking is used on the constrained runtime path, and no concrete call-site evidence was provided to confirm that path exists in this branch.

## Outstanding Concerns

- The branch description and config schema advertise degraded_mode and rerank_enabled as user-facing controls, but they are entirely inert. If these are intentional scaffolding for a future phase, they should be marked as such (e.g., with a comment or a TODO) rather than silently accepting configuration that has no effect.
- The MCP binary's existing current_thread Tokio flavor provides some serialization constraint, but it is not equivalent to the explicit low-CPU runtime the CLI constructs. The asymmetry is undocumented and a user running mempalaced on constrained hardware via MCP has no indication that the low_cpu config section is being ignored by the server process.
- max_blocking_threads = 1 as a low-CPU default warrants verification against actual spawn_blocking usage in the codebase before shipping; if LanceDB or SQLite operations use spawn_blocking under the CLI runtime, a single blocking thread could stall the entire event loop.
