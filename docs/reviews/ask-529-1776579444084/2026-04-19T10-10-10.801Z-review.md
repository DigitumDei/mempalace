## Executive Summary

This branch implements Phase 9 Rust CLI parity for `init`, `mine`, `search`, `status`, and `wake-up`, with `split` and `compress` explicitly deferred. Three consensus blocking defects prevent safe merge: the production CLI constructs `FastembedProvider` without `.try_initialize()`, making `mine`, `search`, and `wake-up` dead on arrival outside stub tests; `init --palace` silently rebuilds `~/.mempalace/config.json` from defaults, wiping any existing `embedding_profile` or other user settings; and `mine --wing` is wired only into `ConversationIngestRequest` and is silently dropped for project ingest, a direct parity failure. The verdict is `revise` because all three defects are confirmed by code inspection, affect core user-visible workflows, and are masked by a test suite that exclusively uses a fake embedding provider.

## Blocking Issues

- Embedding provider never initialized before use (mempalace-rs/crates/mempalace-cli/src/main.rs): `fastembed_provider()` returns `FastembedProvider::new(...)` without calling `.try_initialize()`. `embed()` returns `EmbeddingError::NotInitialized` when `self.backend` is `None`. This makes `mine`, `search`, and `wake-up` fail at runtime. The MCP server already demonstrates the correct pattern with `.try_initialize()`. All CLI tests use `stub_provider` and never exercise this path, so CI is green while the shipped binary is broken.
- `init --palace` overwrites global config with defaults, destroying existing settings (mempalace-rs/crates/mempalace-cli/src/main.rs): `write_global_config_override` constructs `ConfigFileV1 { palace_path: Some(...), ..ConfigFileV1::default() }` and serializes it to disk. Any pre-existing `embedding_profile` or other non-default fields are silently lost. Because the Rust code uses `embedding_profile` to open storage, this can break subsequent commands after what looks like a harmless `init --palace` invocation.
- `mine --wing` silently ignored for project ingest (mempalace-rs/crates/mempalace-cli/src/main.rs): `execute_mine` passes `wing` into `ConversationIngestRequest` but builds `ProjectIngestRequest` with no wing field, so the argument is silently dropped in the default (`--mode projects`) path. The Python reference explicitly supports `wing_override` for project mining. This is a direct CLI parity failure on a core workflow.

## Non-Blocking Issues

- `--yes` flag is advertised but silently ignored (mempalace-rs/crates/mempalace-cli/src/main.rs): `Commands::Init` declares `yes: bool` with help text, but `execute_init` names it `_yes` and takes no action on it. Users who omit the flag get identical behavior to passing it. Either implement an interactive prompt or remove the flag and its help text.
- `mempalace.yaml` unconditionally overwritten on re-init (mempalace-rs/crates/mempalace-cli/src/main.rs): `write_project_config` calls `fs::write` with no prior existence check. Running `init` on a project with an existing hand-curated `mempalace.yaml` destroys it silently.
- `--dry-run` still creates storage files on disk (mempalace-rs/crates/mempalace-cli/src/main.rs): `StorageEngine::open` is called unconditionally before the dry-run branch, which can create SQLite and LanceDB files in an empty palace directory even when the user expects a read-only audit.
- Room detection silently drops secondary directories that map to the same room name (mempalace-rs/crates/mempalace-cli/src/main.rs): `detect_rooms` uses `entry(...).or_insert_with(...)`, so only the first directory normalizing to a given room name is recorded. Subsequent directories mapping to the same room are silently ignored, producing incomplete room descriptions.
- Default agent string changed from `mempalace` to `mempalace-cli` (mempalace-rs/crates/mempalace-cli/src/main.rs): Python writes `mempalace` as the agent name; the Rust CLI defaults to `mempalace-cli`. This is metadata written to drawers and represents parity drift, though it is unlikely to break queries today.
- Exit codes are inconsistent across commands when no palace is found (mempalace-rs/crates/mempalace-cli/src/main.rs): `status` with no palace returns exit 0 (informational); `search` with no palace returns exit 1; config/storage errors exit 2 via the `Err` branch. Scripts relying on exit codes cannot distinguish a missing palace from a configuration error.

## Missing Tests

- Real `FastembedProvider` path: test that `mine`/`search`/`wake-up` fail clearly when `.try_initialize()` has not been called (i.e., a test that would have caught the initialization bug)
- `mine --wing` in project mode: assert that wing is forwarded to `ProjectIngestRequest`
- `init --palace` with existing config: assert that `embedding_profile` and other non-default fields are preserved after the call
- `init` when `mempalace.yaml` already exists: assert the file is not silently overwritten
- `mine --mode convos` through the CLI layer
- `mine --mode convos --dry-run` through the CLI layer
- `--palace <path>` global override flag at the CLI layer
- `search --wing` and `search --room` filtering arguments
- `wake-up --wing` argument

## Disputed Issues

- Environment-sensitivity of `search_without_a_palace_exits_non_zero`: Claude raised that `ConfigLoader::load_with_env` might fall back to `~/.mempalace/palace` in a bare directory, making the test fragile on developer machines; Gemini verified the loader anchors its default palace path to the provided `config_root` override and rejected the concern. The dispute is unresolved in the review record but Gemini's reading aligns better with how override-based config loaders typically behave.

## Outstanding Concerns

- The entire CLI test suite uses `stub_provider`, meaning CI green state provides no signal about whether the real embedding provider lifecycle works. Any future fix to the initialization bug should be accompanied by at least one integration test that exercises `FastembedProvider` end-to-end.
- Whether `ProjectIngestRequest` can structurally accept a `wing` field needs to be confirmed before the `mine --wing` fix is scoped: if the field is absent from the ingest-layer struct, the fix requires changes deeper than CLI wiring.

### Gemini (gemini / gemini-3.1-pro-preview)

```text
Blocking Issues

1. **Destructive Global Config Overwrite in `init`**
When `mempalace-cli init --palace <path>` is executed, the `write_global_config_override` function creates a new `ConfigFileV1::default()` and serializes it, completely overwriting the user's `~/.mempalace/config.json`. This wipes out any pre-existing configuration, such as an `embedding_profile`, LLM settings, or API keys. It must read the existing configuration (if present) and mutate only the `palace_path`.

2. **Destructive `mempalace.yaml` Overwrite and Ignored `--yes` Flag**
The `execute_init` function ignores the `_yes` boolean argument completely. It unconditionally executes `fs::write` on `mempalace.yaml`, overwriting any existing project configuration without a prompt. If `--yes` is omitted, the CLI should either abort, print the plan and instruct the user to run with `--yes`, or provide an interactive prompt. It should never silently destroy an existing user-curated `mempalace.yaml`.

Non-Blocking Issues

1. **`--dry-run` Still Creates Database Files**
During `execute_mine`, `StorageEngine::open` is called before the ingestion logic. If the palace directory doesn't exist, this will initialize an empty SQLite database and LanceDB layout. While it correctly refrains from writing drawers, true dry runs typically avoid modifying the filesystem state at all. 

2. **Room Detection Dictionary Keying**
In `detect_rooms`, the logic maps discovered directories to canonical room names. If multiple folders map to the same room (e.g., `src/frontend` and `src/client` both mapping to `frontend`), `discovered.entry(room_name).or_insert_with` only captures the *first* folder found. The resulting room description and keywords will completely omit the existence of the secondary folders.

3. **`--wing` Silently Ignored for Projects**
The `mine` command accepts a `--wing` argument, but if `mode` is set to `CliMode::Projects` (the default), the argument is silently ignored since `ProjectIngestRequest` doesn't consume it. Clap's `requires` or `required_if_eq` could be used to restrict this flag strictly to `CliMode::Convos`.

4. **GitHub Actions Workflow Placement**
The new `cli-tests` job is added to `.github/workflows/mempalace-rs-storage.yml`. While functional, placing CLI application tests inside a workflow specifically named after the storage crate is misleading. 

Missing Tests

1. `init` command executed with the `--palace` flag (to catch the global config destruction bug).
2. `init` command executed when a `mempalace.yaml` already exists in the project directory.
3. `init` command executed *without* the `--yes` flag (to verify it properly aborts or prompts rather than overwriting data).

Strong Concerns

The `--limit` parameter restricts the number of *files processed* from the discovered set, but it applies this limit *before* checking if the files are unchanged in the database. If a user runs `mine --limit 10` and the first 10 files are already present and unchanged, 0 new files will be ingested. While technically correct for a "processing limit," this could frustrate users who expect `--limit` to guarantee `N` *new* ingestions. Additionally, `summary.discovered_files` logs the total files found *before* the limit is applied, which can make the output slightly confusing to trace.

Confidence

High. The architectural approach for the CLI is very solid, `clap` is leveraged well, the implementation of deferred command stubs is exact, and the E2E tests provide great coverage. However, the data-loss risks introduced by the `init` command must be fixed before this can be safely merged.
```
