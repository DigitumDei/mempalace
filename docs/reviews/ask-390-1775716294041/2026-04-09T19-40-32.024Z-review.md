## Executive Summary

This branch establishes a Phase 0 reference baseline for a Python-to-Rust migration by capturing CLI outputs, MCP schemas, search behavior, wake-up behavior, and other key surfaces as committed fixtures with drift checks. All three reviewers reached strong consensus that the drift workflow has multiple hard defects making it non-functional: the capture script crashes immediately with a NameError before producing output, the drift script's inputs/ exemption guard is unreachable so every clean run exits non-zero with spurious failures, and the committed fixture-lock.json is already inconsistent with what the current capture script would produce. The wake-up drift comparator adds a fourth blocking issue by comparing only structure in the regenerated output while ignoring the baseline content, providing near-zero regression signal. These failures strike at the core value proposition of the change and the verdict is revise.

## Blocking Issues

- NameError: FIXTURE_ROOT undefined in phase0_capture.py (scripts/phase0_capture.py): scripts/phase0_capture.py defines SOURCE_FIXTURE_ROOT but references the undefined name FIXTURE_ROOT when building input_hashes. This causes an immediate NameError before the lock file is written, making the documented Phase 0 regeneration workflow completely non-functional.
- inputs/ exemption guard is dead code in check_phase0_drift.py (scripts/check_phase0_drift.py): before is loaded from the committed fixture tree (which includes inputs/ files) while after is loaded from the temp output tree (which never contains inputs/). For every inputs/* path, the rel not in after branch fires and appends a failure before the startswith('inputs/') exemption is reached. Every drift-check run exits non-zero with four spurious 'file added or removed' failures regardless of whether behavior actually changed.
- Committed fixture-lock.json is inconsistent with current capture script (tests/fixtures/phase0/fixture-lock.json): The committed fixture-lock.json includes inputs/ paths in generated_hashes, but the current capture script builds generated_hashes by iterating only GOLDEN_ROOT and INVENTORY_ROOT. Once the NameError is fixed, regenerating the lock will produce a file that diverges from the committed one, causing test_phase0_fixture_lock_matches_files to fail. The committed artifact was generated by an earlier version of the script.
- _compare_wake_up ignores before["rooms"] entirely (scripts/check_phase0_drift.py): The function only verifies that the regenerated output's header matches and that after["rooms"] is non-empty with names and bullets. The baseline rooms are never consulted. A regenerated wake-up could return completely different room names and content and pass. The test suite explicitly codifies this broken behavior as acceptable (replacing [auth-migration] with [release-readiness] asserts a pass), making this a documented false-passing case rather than a latent risk.

## Non-Blocking Issues

- search-cli.txt has contradictory exact/tolerant policy across scripts (scripts/check_phase0_drift.py): phase0_capture.py and the lock metadata classify search-cli.txt as TOLERANT_OUTPUTS, excluding it from exact hashes. check_phase0_drift.py places it in EXACT_FILES, requiring byte-for-byte comparison. The file contains floating-point similarity scores that are unlikely to be bit-identical across chromadb versions or hardware. The two scripts have picked opposite policies for the same artifact.
- _compare_programmatic_search enforces exact ordering, contradicting the parity matrix (scripts/check_phase0_drift.py): The comparator uses zip(before_results, after_results), enforcing position-by-position identity. The parity matrix states 'exact ordering only where scores tie.' Valid backend tolerance causing two results to legitimately swap positions will produce a false drift failure. The test suite confirms strict ordering was intentional, making this a documented spec-vs-implementation divergence.
- test_phase0_drift_script_does_not_modify_workspace never runs the drift script (tests/test_phase0_assets.py): The test only manipulates in-memory dictionaries and asserts against a mock failure; it never calls check_phase0_drift.main() or spawns the script as a subprocess. The test name implies it verifies workspace safety, but the actual script is never executed.

## Missing Tests

- End-to-end test that runs phase0_capture.py successfully and verifies exit 0 — would have caught the FIXTURE_ROOT NameError immediately
- Test that runs check_phase0_drift.py (or main()) on a freshly captured temp output and asserts exit 0, without silently skipping when chromadb is unavailable
- Negative test proving _compare_wake_up rejects a payload where a required room goes missing or bullet content substantially changes
- Test verifying that regenerating the capture produces a fixture-lock.json whose generated_hashes matches the committed lock — directly exposes the current inconsistency
- Negative drift-detection test proving the drift script actually exits non-zero when a committed golden (e.g., goldens/mcp-contract.json) is mutated

## Disputed Issues

- Severity of _compare_wake_up weakness: Codex classifies it blocking, Claude and Gemini treat it as a design defect that provides weak signal without crashing — all agree it must be fixed, disagreement is only on severity label
- pytest plain-invocation collection failure: Codex raised it, Claude and Gemini considered it too weak to list as a finding since it does not affect the documented CI path

## Outstanding Concerns

- The integration test test_phase0_drift_script_is_stable_when_vendor_env_exists silently returns without asserting if chromadb is not importable from .phase0_vendor/. This means the NameError and dead-code bugs in the drift workflow are currently invisible in any CI environment where chromadb is unavailable, giving a false green on the core Phase 0 safety check.
- Zero-network replay is documented as a known limitation deferred to before Phase 1, but the Chroma/ONNX first-run bootstrap still requires network access. If the Phase 0 baseline is intended to be reproducible offline, this gap must be resolved before the baseline can serve as an authoritative reference for Rust parity validation.

### Gemini (gemini / gemini-3.1-pro-preview)

```text
**Blocking Issues**

- **Wake-up Tolerance Logic is Broken**: The `_compare_wake_up` function in `check_phase0_drift.py` is structurally flawed. It only checks that the header matches and that the new output contains *some* rooms and bullets (`all(room["room"] and room["bullets"] for room in after_rooms)`). It completely ignores the contents of `before["rooms"]`. This means the semantic content of the wake-up could drastically change or disappear and the drift check would still pass. Paradoxically, `test_phase0_wake_up_tolerance_requires_structure` explicitly asserts this broken behavior by proving that changing `[auth-migration]` to `[release-readiness]` passes the check.
- **Contradictory Exact vs Tolerant File Definitions**: There is a fundamental mismatch between the capture script and the drift script. `phase0_capture.py` treats `search-cli.txt` as a `TOLERANT_OUTPUT` (excluding it from the exact hashes), but `check_phase0_drift.py` categorizes it under `EXACT_FILES`. Conversely, `wake-up.txt` and `wake-up-wing-code.txt` are hashed as exact in the capture script, but are treated as `TOLERANT_FILES` in the drift checker.
- **`search-cli.txt` Treated as Exact**: Because `check_phase0_drift.py` includes `search-cli.txt` in `EXACT_FILES`, it forces a byte-for-byte comparison. However, this file contains similarity scores (`Match: 0.49`, etc.). The Parity Matrix explicitly allows numeric values to differ within model tolerance. Valid float drift in the embedding backend will cause this exact check to fail.
- **Programmatic Search Tolerance Enforces Exact Ordering**: `_compare_programmatic_search` evaluates lists by tightly coupling them with `zip(before_results, after_results)`. The Parity Matrix specifies "exact ordering only where scores tie", but this zip approach demands exact ordering regardless of scores. If valid backend tolerance causes two results to legitimately swap places, the drift check will falsely fail.

**Non-Blocking Issues**

- **Brittle Environment Inventory Drift Check**: By treating `inventory/environment.json` as an `EXACT_FILE`, the drift check will fail if the script is run on a machine with Python 3.11.3 instead of 3.11.2, or if a minor patch updates a dependency. While the reference environment should be locked, failing the behavioral drift suite over local environment patch-versions makes the tool highly frustrating for developers to run locally.
- **Unsorted Programmatic Search Golden**: Unlike the MCP search payload (which uses `_normalize_search_results` to sort results deterministically), the results for `search-programmatic.json` are dumped directly from `search_memories()` into the golden. Explicitly sorting these results before writing would make the baseline significantly more stable across platforms.
- **Temp Directory Leak Risk**: `phase0_capture.py` relies on `tempfile.mkdtemp()` and cleans it up via `shutil.rmtree` in a `finally` block. If the process is forcefully killed (SIGKILL) or crashes at a C-extension level, the directory leaks in `/tmp`. Utilizing `tempfile.TemporaryDirectory()` as a context manager is safer and more idiomatic.

**Missing Tests**

- **Tolerant Search Failure Modes**: There are no tests verifying that `_compare_programmatic_search` actually fails if the returned text content is different, or if the similarity drifts *beyond* the 0.05 tolerance limit.
- **Wake-Up Semantic Drift**: Missing tests to prove that `_compare_wake_up` correctly rejects a payload when a required room goes missing or when the textual content of a bullet significantly deviates from the golden.
- **Unmocked E2E Drift Verification**: There is no un-skipped integration test that runs `phase0_capture.py` and then unconditionally verifies `check_phase0_drift.py` passes immediately after on the real workspace (the existing `test_phase0_drift_script_is_stable_when_vendor_env_exists` silently returns without failing if `.phase0_vendor` is absent).

**Strong Concerns**

- **Zero-Network Replay Vulnerability**: The `reference-environment.md` states that Chroma's implicit embeddings still require network access to bootstrap the model on the first run, and defers fixing this to before Phase 1. If this is truly a "Phase 0 locked baseline", relying on a dynamic network fetch for the ONNX model undermines the concept of a reproducible offline environment. Pinning the local model asset path should be resolved before merging this freeze.

**Confidence**
High. The implementation directly contradicts its own design documents (Parity Matrix) regarding search ordering and tolerances, and the wake-up verification logic contains a glaring omission of the baseline variable.
```
