## Executive Summary

This branch establishes a Phase 0 baseline for a Rust migration parity effort by capturing Python reference outputs, documentation, and drift-detection tooling. All three reviewers converged on two critical defects: the drift-check script regenerates fixtures in-place without a temporary directory or restore step, turning a safety check into a destructive auto-updater that silently clears drift evidence on a second run; and the parity enforcement does not match the documented parity matrix, with wake-up goldens enforced as byte-exact while the spec classifies them tolerant, and CLI search formatting enforced as tolerant while the spec requires it exact. These contradictions undermine the stated goal of a spec-locked baseline and must be resolved before Phase 1 can safely begin.

## Blocking Issues

- Drift check rewrites committed fixtures in place (scripts/check_phase0_drift.py): check_phase0_drift.py invokes phase0_capture.py which writes regenerated outputs directly into FIXTURE_ROOT before comparison. If drift is detected, the original committed goldens are destroyed on disk (recoverable only via git). Worse, running the script a second time regenerates from the same environment, compares already-overwritten files to themselves, and silently passes — hiding the original drift. The script must write to a temporary directory and compare against the committed tree, or restore the before-state after execution.
- Wake-up goldens in EXACT_FILES conflict with tolerant parity classification (scripts/check_phase0_drift.py): parity-matrix.md classifies wake-up drawer selection as Tolerant Parity because drawer selection depends on similarity scores that vary with embedding backend. reference-environment.md admits the Chroma embedding model may warm from the network on first run and that zero-network replay is unproven. Despite this, check_phase0_drift.py lists wake-up.txt and wake-up-wing-code.txt in EXACT_FILES, enforcing byte-for-byte matching. The drift check will spuriously fail in any environment whose embeddings differ from the capture environment.
- CLI search formatting treated as tolerant while parity matrix requires exact (scripts/check_phase0_drift.py): parity-matrix.md places 'Search formatting shape' — explicitly including the Match label — in the Exact Parity table. check_phase0_drift.py places search-cli.txt in TOLERANT_FILES, and the _parse_search_cli normalizer unconditionally strips all Match: lines and sorts result blocks. A Rust implementation that drops score lines entirely or reverses ranking would pass the drift check despite violating the documented exact-parity contract.

## Non-Blocking Issues

- _run_help() ignores subprocess exit status (scripts/phase0_capture.py): phase0_capture.py calls help commands with check=False and records only stdout. If a help command fails and writes a traceback to stderr, the golden silently records an empty or partial string with no error raised, producing a corrupt baseline with no signal.
- Crash on missing tolerant file in drift check (scripts/check_phase0_drift.py): If a tolerant file is added or removed, the first loop in check_phase0_drift.py appends to failures but does not exclude the path from the second loop. The second loop calls _compare_tolerant on the missing path, raising FileNotFoundError instead of reporting a clean failure.
- Drift script integration test silently skips in CI (tests/test_phase0_assets.py): test_phase0_drift_script_is_stable_when_vendor_env_exists returns early when .phase0_vendor/ is absent, which is the case in any clean checkout. The most important integration test — confirming the drift script confirms stability — never executes in CI, so the drift infrastructure has no automated verification of its own correctness.
- Reference environment uses floating package versions (docs/rust-phase0/reference-environment.md): The documented bootstrap command in reference-environment.md installs packages without pinned versions. The fixture lock records what was resolved after the fact but does not make the environment reproducible from scratch. A future capture on a different machine may silently produce different outputs.

## Missing Tests

- No negative test that deliberately perturbs a golden and confirms check_phase0_drift.py exits non-zero — the drift machinery could silently always pass and no automated test would catch it
- No test asserting that running the drift check leaves the working tree unmodified
- No test validating ranking regression is caught — the current normalizer sorts results alphabetically, so reversed ordering passes

## Disputed Issues

- Whether search ranking and similarity preservation are blocking for Phase 0: Codex treats the ranking normalizer as blocking because it weakens the parity contract; Claude argues Phase 0 is a Python-stability baseline and Rust parity enforcement is a Phase 1 concern, making this at most a non-blocking gap in tolerant-contract definition
- KnowledgeGraph() path leak: Claude flagged it as requiring verification; Codex assessed it as invalid given that HOME is overridden before KnowledgeGraph is imported and the default path resolves at import time

## Outstanding Concerns

- Zero-network embedding gap combined with exact-match wake-up goldens makes Phase 1 prerequisites contradictory: reference-environment.md explicitly states Phase 0 does not yet prove zero-network replay for Chroma's implicit embedding model, but wake-up goldens are in EXACT_FILES; Phase 1 cannot safely start until this is resolved
- fixture-lock.json version field is never validated by either the drift check or the test suite; a format change would silently produce wrong comparison results rather than a fast failure
