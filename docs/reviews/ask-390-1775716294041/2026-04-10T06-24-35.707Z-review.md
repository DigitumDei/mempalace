## Executive Summary

This branch establishes a Phase 0 spec-lock baseline for a Python-to-Rust migration by capturing CLI output, MCP schemas, search behavior, wake-up output, and graph artifacts as committed reference fixtures, along with scripts to regenerate and drift-check them. All three reviewers independently confirmed two structural defects that make the baseline non-functional as a contract enforcement tool: the committed `fixture-lock.json` includes `inputs/` entries under `generated_hashes` that the current capture script never writes, so the first legitimate regeneration will mutate an exact-match artifact and break drift detection immediately; and the `_compare_wake_up` comparator is asymmetric, checking only that new output is a subset of the baseline, meaning a Rust implementation emitting a single room and bullet passes the parity gate. Until these two issues are fixed, the Phase 0 artifact set cannot be regenerated or validated as documented, which directly contradicts the branch's stated goal of a trustworthy, reproducible spec-lock. The verdict is revise.

## Blocking Issues

- fixture-lock.json generated_hashes includes inputs/ entries that the capture script never produces (tests/fixtures/phase0/fixture-lock.json): phase0_capture.py computes generated_hashes by iterating only (GOLDEN_ROOT, INVENTORY_ROOT), but the committed fixture-lock.json contains four inputs/… entries under generated_hashes. Because fixture-lock.json is in EXACT_FILES, running the capture script fresh produces a lockfile that differs byte-for-byte from the committed one, causing drift detection to fail on its own output. The test test_phase0_fixture_lock_matches_files compounds this by hashing all non-tolerant files under the fixture root (including inputs/), encoding the wrong invariant. The committed artifact was generated by a different version of the script than the one in this branch.
- _compare_wake_up is asymmetric: dropped rooms and bullets pass drift detection (scripts/check_phase0_drift.py): The comparator loop at check_phase0_drift.py:162 iterates after_rooms and checks membership in before_rooms, but never checks the reverse. Rooms present in the baseline that are absent from the regenerated output are invisible to the gate. The test test_phase0_wake_up_tolerance_requires_structure explicitly asserts a 10-line truncated output passes, confirming the subset-only behavior is baked into the test contract. A Rust implementation emitting a single room and one bullet would pass this parity gate, which contradicts the stated acceptance criterion of preserving top-story usefulness.

## Non-Blocking Issues

- search-cli.txt is in EXACT_FILES but contains volatile float similarity scores (tests/fixtures/phase0/goldens/search-cli.txt): search-cli.txt hardcodes Match: 0.49, Match: 0.069, Match: -0.267 and is classified as exact-match, while search-programmatic.json for the same surface is in TOLERANT_FILES with a 0.05 tolerance gate specifically because similarity values drift with model or backend updates. A minor ChromaDB or embedding model version bump will break the exact CLI contract while the programmatic contract remains stable. The two files should use consistent classification.
- Programmatic search drift check is order-agnostic, not order-tolerant (scripts/check_phase0_drift.py): check_phase0_drift.py lines 85–107 build identity-keyed dicts and compare only the key sets, ignoring result order entirely. A top-hit swap with a lower-ranked result passes silently. The test at test_phase0_assets.py:186 explicitly accepts a fully reversed result list and asserts true. This is not 'tolerant of minor similarity drift' but 'completely agnostic to ranking', which is weaker than the docs claim.
- _parse_wake_up returns False for two structurally identical short outputs (scripts/check_phase0_drift.py): Outputs shorter than 6 lines produce empty room lists. _compare_wake_up then returns False because of the guard if not before_rooms or not after_rooms, incorrectly reporting drift between two outputs that are structurally identical. Benign in practice but mechanically incorrect.
- Python patch version embedded in EXACT_FILES causes cross-environment failures (tests/fixtures/phase0/fixture-lock.json): environment.json and fixture-lock.json embed sys.version.split()[0] (e.g., 3.11.2). Any run on a different patch level triggers an exact-match drift failure, making local developer validation brittle and CI matrix runs across Python 3.9/3.11/3.13 inherently fragile.
- test_phase0_drift_script_is_stable_when_vendor_env_exists silently no-ops in CI (tests/test_phase0_assets.py): .phase0_vendor is untracked and not committed. In a clean CI environment the test hits if not vendor.exists(): return and exits without testing anything, leaving no integration coverage of the drift check script's actual end-to-end behavior.

## Outstanding Concerns

- The MCP search contract golden (goldens/mcp-contract.json) is a normalized projection of Python behavior (similarity stripped, results re-sorted) rather than a raw observable output snapshot. While normalization is standard practice for volatile fields, the gap between 'what Python actually returns' and 'what the golden captures' is undocumented in the parity matrix and weakens the contract claim.
- KnowledgeGraph() isolation: the capture script sets HOME before the deferred import, which likely works for the default ~/.mempalace path, but the isolation guarantee is not tested and would silently fail if the KnowledgeGraph implementation resolves its storage path at module load time rather than at construction time.
