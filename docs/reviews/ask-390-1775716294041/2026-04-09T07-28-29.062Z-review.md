## Executive Summary

This branch establishes a Phase 0 spec-lock for the Rust port of mempalace by capturing Python reference behavior as golden assets, adding drift-detection tooling, and documenting parity targets. All three reviewers converge on a fundamental structural defect: the drift checker is broken from day one because the committed MCP golden embeds a non-deterministic mkdtemp path, ensuring every recapture produces a different SHA256 hash. Beyond that single failure, the drift checker's uniform byte-exact strategy is architecturally incompatible with the parity matrix's own two-tier contract, which explicitly marks search ranking and similarity scores as tolerant — meaning even a correctly regenerated MCP golden would still yield false-positive drift failures on search outputs. Additionally, the corpus is incomplete against its own acceptance criteria: the knowledge-graph `invalidate` golden required by acceptance-criteria.md line 31 is never captured, and the test suite's assertions are too shallow to enforce any of the above. The branch has the right structure and documentation intent but requires targeted fixes before the baseline can function as a reliable regression tool.

## Blocking Issues

- Non-deterministic palace_path embedded in committed MCP golden breaks drift checker permanently (tests/fixtures/phase0/goldens/mcp-contract.json): scripts/phase0_capture.py creates a fresh mkdtemp path on every run and exports it as MEMPALACE_PALACE_PATH; mcp-contract.json line 50 commits the literal temp path. check_phase0_drift.py compares SHA256 hashes, so every recapture produces a different hash regardless of behavioral change, making the drift checker permanently non-functional as a regression tool.
- Byte-exact drift check applied to search goldens contradicts parity matrix's tolerant-parity contract (scripts/check_phase0_drift.py): parity-matrix.md lines 23–25 explicitly designate ranking and similarity scores as requiring only tolerant parity, yet check_phase0_drift.py applies a single SHA256 strategy to all files including search-programmatic.json which embeds floating-point similarity scores. The drift checker has no tolerance mode, so any benign model or backend drift in search scores registers as a regression, making the checker useless as a parity oracle for the Rust port.
- Knowledge-graph `invalidate` golden missing despite being required by acceptance criteria (scripts/phase0_capture.py): acceptance-criteria.md line 31 explicitly lists invalidate as a required golden operation. The capture script (lines 254–261) never calls kg.invalidate() and the committed knowledge-graph.json contains only query, timeline, and stats entries. The Phase 0 corpus is incomplete against its own stated contract.

## Non-Blocking Issues

- `connected_via` field committed to in parity matrix never appears in palace-graph golden (tests/fixtures/phase0/goldens/palace-graph.json): parity-matrix.md line 19 promises connected_via preservation in graph output. The committed palace-graph.json only captures hop-0 traversal; connected_via only appears in palace_graph.py inside the depth+1 branch. Future Rust implementors have no reference for what this field must look like.
- Reference environment doc records `python3` instead of a pinned version (docs/rust-phase0/reference-environment.md): acceptance-criteria.md requires the reference-environment doc to record the Python version used. reference-environment.md line 5 only states 'Python executable: python3'; the specific version is captured in environment.json but absent from the human-readable reference doc.
- environment.json full sys.version string (including compiler date) will cause spurious drift across machines (tests/fixtures/phase0/inventory/environment.json): environment.json stores the full Python sys.version string including compiler details and build dates (e.g. '3.11.2 (main, Apr 28 2025, 14:11:48) [GCC 12.2.0]'). Any developer running on a different Python build triggers a hash mismatch unrelated to behavioral change.
- Inventory records ambient installed packages rather than declared dependency inputs (scripts/phase0_capture.py): reference-environment.md describes the inventory as derived from pyproject.toml and requirements.txt, but the capture script (line 197) uses importlib.metadata.version() on the running interpreter, recording whatever is ambient. This makes the inventory more machine-specific than the documentation implies.

## Missing Tests

- No test validates that fixture-lock.json hash values match actual file contents — the suite checks key shape only, not hash correctness, so any manually edited or diverged golden passes silently
- No test asserts that the eight golden files, three inventory files, or the full expected output set exist — a missing golden is invisible to the current test suite
- No test exercises check_phase0_drift.py or phase0_capture.py — the central regression tool has zero coverage and its broken behavior goes undetected
- No test covers the combined wing+room `$and` search case, which parity-matrix.md line 12 marks as exact-contract behavior
- No test parses or validates the structural correctness of JSON goldens — truncated or structurally wrong goldens (especially mcp-contract.json as the primary contract artifact) pass the suite undetected

## Disputed Issues

- os.environ['HOME'] mutated 13 lines before the try block (phase0_capture.py line 75 vs line 88): Claude rated this blocking due to potential process-state corruption; Codex rated it overstated given the low-risk code between mutation and try. Not included as a blocking consensus item.
- ChromaDB float non-determinism across hardware architectures (AVX vs NEON): Gemini raised as a strong concern; Codex called it speculative. The concern is valid in principle but the branch does not claim cross-architecture reproducibility.
- Windows line-ending sensitivity in write_text output: Gemini raised; Codex dismissed as unsupported given the project makes no Windows portability claim.
- Hand-seeded corpus not fully derived from committed fixture inputs (capture script lines 110–157 call col.add() with hardcoded documents not in tests/fixtures/phase0/inputs/): Codex raised as strong concern; not elevated to blocking by consensus but represents a reproducibility assumption that should be documented explicitly.

## Outstanding Concerns

- The drift checker is architecturally mismatched with the parity matrix's two-tier (exact vs. tolerant) contract. Fixing the temp-path embedding alone is insufficient — check_phase0_drift.py needs a tolerance mode for search similarity scores and a strategy for volatile environment metadata, or the matrix's tolerant-parity surfaces must be excluded from byte-exact comparison. Without this redesign, the drift checker will continue to report false positives even after individual defects are fixed.
- The capture corpus is partly hand-seeded rather than fully derived from committed fixture inputs. Goldens that depend on in-script col.add() data cannot be independently reproduced or verified from source alone. This should be explicitly documented in the Phase 0 spec as a known constraint, or the fixture inputs should be updated to include the seeded data.
