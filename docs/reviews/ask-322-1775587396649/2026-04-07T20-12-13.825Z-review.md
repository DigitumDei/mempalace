## Executive Summary

This branch adds two new planning documents — docs/RustMigration.md and docs/RustMigrationTasks.md — that lay out a subsystem-by-subsystem comparison of the current Python/Chroma implementation against a proposed Rust rewrite using LanceDB, SQLite, and explicit local embeddings. The documents are technically substantive and architecturally coherent, but they contain two material planning failures that prevent them from being published as authoritative migration guidance. Phase 10 labels itself a 'safe path to Rust' while only scoping Chroma-to-LanceDB vector conversion, leaving config.json, people_map.json, entity_registry.json, onboarding-generated artifacts, and the SQLite knowledge graph entirely unaddressed. Separately, the contract-test and golden-fixture strategy in RustMigrationTasks.md is built on Python outputs whose reproducibility is never guaranteed, because the Python environment, Chroma version, and ONNX model cache are never pinned — a gap the migration doc itself acknowledges on the subject of implicit embedding behavior. Until both gaps are resolved the plan will mislead implementers about migration safety and test reliability.

## Changed Files

- docs/RustMigration.md
- docs/RustMigrationTasks.md

## Blocking Issues

- Phase 10 migration scope omits most persisted Python user state (docs/RustMigrationTasks.md): RustMigrationTasks.md Phase 10 operationalizes only Chroma-to-LanceDB vector conversion and metadata remapping, yet a live Python installation carries additional load-bearing local state: ~/.mempalace/config.json and people_map.json (config.py:81-82), entity_registry.json with its five-field structure (entity_registry.py:272/294), onboarding-generated aaak_entities.md and critical_facts.md (onboarding.py:436/445-446), and the SQLite knowledge graph at ~/.mempalace/knowledge_graph.sqlite3 (knowledge_graph.py:46). Phase 10's exit criteria test only 'data-count parity' and 'metadata parity', which presupposes Chroma data is the only thing being migrated. An implementer following the plan as written will ship a migration tool that silently drops the entity registry, people map, config, and knowledge graph. The 'safe path' claim is unsupported by the plan's actual scope.
- Golden fixture and contract-test strategy lacks a pinned Python reference environment (docs/RustMigrationTasks.md): RustMigrationTasks.md Phase 0 'Spec Lock' and subsequent parity phases require frozen Python golden outputs for retrieval and wake-up shapes (lines 41-60, 319-340). RustMigration.md at line 214-224 explicitly acknowledges that Chroma's default embedding behavior is implicit and version-sensitive — the app never sets an embedding function in searcher.py:15 or miner.py:183, and pyproject.toml does not pin chromadb tightly. The task plan never specifies a locked chromadb version, a pinned ONNX model asset, or a documented offline/warm-cache procedure for regenerating fixtures. Regenerating goldens after any Python dependency upgrade will produce different embeddings, different distances, and different rankings. The entire contract-test strategy is therefore non-reproducible as specified, which invalidates the TDD foundation the plan claims to establish.

## Non-Blocking Issues

- Workspace crate layout is inconsistent between the two documents (docs/RustMigration.md): RustMigration.md line 92 proposes 8 crates with config folded into mempalace-core. RustMigrationTasks.md line 285 expands to 11 crates, adding mempalace-config, mempalace-embeddings, and mempalace-migrate as separate crates. An implementer reconciling the two will have to make an architectural decision the docs should have made. This is a quality issue, not a safety one.
- `split` CLI command inventoried but never scheduled in execution plan (docs/RustMigrationTasks.md): RustMigration.md line 596 lists `split` as part of the current Python CLI surface, and the command is real at cli.py:121 backed by split_mega_files.py. Phase 9 of RustMigrationTasks.md covers CLI parity (lines 621-651) but never mentions `split` and never states it is intentionally dropped. The parity claim is incomplete.
- Phase 2 storage schema implies embedding dimension before Phase 3 pins the model (docs/RustMigrationTasks.md): Phase 2 defines DrawerRecord with embedding: Vec<f32> and creates the LanceDB table, which requires a fixed-width schema at creation time. Phase 3 is where the model and therefore the embedding dimension is locked. The plan does not call out this ordering dependency, and the dimension constant is never declared as a Phase 1 commitment. An implementer doing Phase 2 in isolation will either hardcode 384 as a hidden constraint or leave a placeholder that breaks at Phase 3.

## Missing Tests

- Migration acceptance tests for non-Chroma persisted state: config.json, people_map.json, entity_registry.json, aaak_entities.md, critical_facts.md, and knowledge_graph.sqlite3 — Phase 10 test coverage is currently scoped only to Chroma vector data.
- Environment-lock fixture-generation contract for Python golden outputs: pinned chromadb version, pinned ONNX model artifact, and a documented offline/warm-cache procedure for regenerating retrieval and wake-up snapshots reproducibly.
- Fixture drift detection test: a mechanism for detecting when Python fixture regeneration produces different output due to an embedding version or dependency change, so the contract test baseline does not silently shift.

## Disputed Issues

- Whether the Chroma 'practical default today is local all-MiniLM-L6-v2' claim is accurate enough to publish: Codex (round 1) called it too strong to publish as fact; Claude and Gemini countered that the surrounding text already qualifies it as implicit and version-sensitive, and that the repo's own benchmarks establish the same baseline. Resolved toward non-blocking by round 2 consensus.
- Whether listing 'MCP exposure to hosted LLM clients' under 'What does not automatically improve' fails the stated success criterion to reflect an accepted risk model: Gemini raised this as blocking; Codex and Claude found the text factually neutral and found no cited requirement mandating specific framing language. Not resolved — Gemini's criterion was never quoted from the source materials.

## Outstanding Concerns

- The contract-test plan implicitly requires maintaining a working Python reference environment with pinned dependencies and a warm ONNX model cache in CI indefinitely. For a project whose stated goal is to move off Python, this is a non-trivial long-term commitment that the plan does not acknowledge or budget for.
- The `rmcp` crate is named as the Rust MCP implementation without validation of its maintenance status or API stability. The plan uses 'rmcp or another Rust MCP implementation' hedging language, but no fallback evaluation criteria are provided for one of the most user-visible integration points.
