## Executive Summary

This branch adds two documentation files planning a Rust rewrite of MemPalace, covering architecture, module-by-module analysis, embedding strategy, e2-micro deployment constraints, phased tasks, and a TDD test plan. The reviewers reached strong consensus on two blocking issues: the proposed canonical `DrawerRecord` schema omits `date` and ranking/weight fields (`importance`, `emotional_weight`, `weight`) that current Python code actively reads, creating an internal contradiction between the storage spec and the parity/Layer 1 ranking promises made elsewhere in the same document; and the dual-LanceDB-plus-SQLite write path has no stated write order, transaction boundary, or crash-recovery strategy, meaning the idempotency and deduplication guarantees the plan claims are unsupported by the design. Several non-blocking gaps compound these: missing modules from the inventory, a hollow TDD framing that names coverage categories without measurable acceptance criteria, and missing runtime cost disclosure for `fastembed`/ONNX on constrained hardware. The verdict is revise until the schema and consistency issues are resolved.

## Blocking Issues

- DrawerRecord schema omits `date` and ranking/weight metadata (docs/RustMigration.md): The proposed canonical schema at RustMigration.md:155–173 lacks `date` and any weighting field (`importance`, `emotional_weight`, `weight`). palace_graph.py:55 reads `date` per record; layers.py:115–128 reads all three weight fields for Layer 1 scoring. The same document promises graph parity and richer Layer 1 ranking, so the storage spec directly contradicts the behavior commitments made elsewhere. A Rust Layer 1 ranker built from this schema would have no field to sort on.
- Dual-write between LanceDB and SQLite has no consistency or crash-recovery strategy (docs/RustMigration.md): The architecture splits drawer/vector writes to LanceDB and manifest/file-hash/operational-state writes to SQLite with no stated write order, transaction boundary, or reconciliation path. A crash between the two writes leaves the system in an inconsistent state. The plan claims idempotency and deduplication correctness, but those properties cannot be guaranteed without a defined recovery strategy for orphaned records on either side.

## Non-Blocking Issues

- Three real modules omitted from architecture inventory (docs/RustMigration.md): room_detector_local.py, split_mega_files.py, and spellcheck.py all exist and the first two are wired into named CLI subcommands. A developer working solely from this document will miss them. room_detector_local.py is particularly significant because the doc describes room detection only as heuristic routing inside miner.py.
- Chunker described as crude/char-count when it is paragraph-aware (docs/RustMigration.md): miner.py:138–164 tries \n\n then \n boundaries before falling back to raw character position. The doc calls it 'char-count based' and 'crude', which understates the current behavior and could cause the Rust replacement to be underengineered.
- fastembed ONNX Runtime footprint not disclosed for e2-micro targets (docs/RustMigration.md): The doc recommends fastembed as the initial embedding backend and extensively discusses CPU budgets, but never flags that fastembed bundles the ONNX Runtime shared library. On an e2-micro this is a meaningful binary and memory dependency that could surprise an implementor targeting the low_cpu profile.
- rusqlite synchronous driver incompatible with LanceDB's Tokio async runtime (docs/RustMigration.md): LanceDB runs on Tokio; blocking rusqlite calls on the async executor degrade latency and can starve the runtime. The doc presents rusqlite vs sqlx as a preference tradeoff without noting this concrete failure mode. sqlx or tokio-rusqlite should be the default recommendation.
- Wikipedia lookup attributed to wrong module (docs/RustMigration.md): The doc names entity_detector.py as the module responsible for optional Wikipedia lookup, but the lookup lives in entity_registry.py. A Rust author working from this document will look in the wrong place first.
- LanceDB embedding dimension mismatch failure mode not explained (docs/RustMigration.md): The doc mentions locking dimensions in Phase 1 but does not state that a mismatch between stored vector dimensions and query vector dimensions is a hard failure, not graceful degradation. An implementor switching embedding profiles on an existing corpus without a migration will encounter a silent or loud breakage with no warning from the spec.
- Embedding behavior described as contract when it is inference from Chroma defaults (docs/RustMigration.md): searcher.py:38 and mcp_server.py:188 never set an explicit embedding function. Any statement in the doc stronger than 'observed Chroma default behavior in a pinned environment' is inference, not an app contract. The distinction matters for parity claims.

## Missing Tests

- No quantified performance targets for low_cpu benchmark gates — 'threshold' and 'budget' appear in Phase 3 and Phase 11 but no actual numbers are committed, making the definition-of-done unverifiable
- No test accounting for floating-point and tokenizer differences between Python Chroma golden fixtures and Rust fastembed outputs — the plan requires exact snapshot parity for retrieval tests but does not allow for fuzzy matching, distance tolerances, or set-overlap metrics
- No cross-database consistency tests simulating crashes during the dual LanceDB+SQLite write to verify recovery from inconsistent state
- No test for concurrent MCP writes (e.g., add_drawer) during an active miner ingest
- No explicit test coverage for the compressed drawers secondary Chroma collection during Phase 10 import/interop
- No fixture or contract tests for entities.json, identity.txt, or legacy mempal.yaml if these are in scope for optional import
- No test for the split_mega_files behavior — whether in or out of scope for the Rust version is never stated

## Disputed Issues

- Whether omission of entities.json, identity.txt, and mempal.yaml from the interop scope is a blocking contradiction (Codex) or an explicit scoping decision that is non-blocking (Claude, consensus critique) — the doc at line 649–651 explicitly narrows import commitment, making this a scope choice rather than an oversight
- Filename RustMigration.md vs RustRewrite.md — Gemini raised this as contradicting user intent, but no in-repo evidence was cited and the critique panel declined to validate it without a source

## Outstanding Concerns

- TDD framing is partially hollow: RustMigrationTasks.md claims no production module starts without tests and acceptance criteria already written, but the suite as written defines coverage categories and process, not measurable acceptance criteria. A developer reading this as a strict TDD contract will find the hard specification work deferred rather than front-loaded.
- Phase 0 fixture-generation infrastructure is called for (pinned reference environment, model asset checksums, offline warm-cache regeneration) but no guidance is given on how it should be managed. On a project targeting e2-micro deployments, initial fixture generation may require a heavier machine, and leaving this undesigned floats the entire test strategy that depends on it.
- people_map.json is listed for optional import in Phase 10 but the Rust onboarding and entity design sections never acknowledge it as a data source. If the Rust onboarding module ignores it, users who go through the optional import path will have a silent registry gap.

### Gemini (gemini / gemini-3.1-pro-preview)

```text
Blocking Issues

- **Split-Brain Dual-Write Problem:** The architecture proposes splitting the storage layer between `LanceDB` (drawers/vectors) and `SQLite` (operational state, manifests, graph). This introduces a classic dual-write vulnerability. The design document lacks a strategy for transactional integrity or crash recovery. If a write to LanceDB succeeds but the subsequent write to SQLite fails (or vice versa), the system state will be corrupted. The doc must define the write order and a reconciliation or cleanup strategy for orphaned records.

Non-Blocking Issues

- **Sync vs. Async DB Drivers:** The document suggests `rusqlite` is "enough unless you expect heavy async server concurrency". However, `LanceDB` is fundamentally async and runs on Tokio. Using the synchronous `rusqlite` crate alongside LanceDB means database calls will either block the async executor (bad performance) or require `spawn_blocking` wrappers (added complexity). Recommending `sqlx` or `tokio-rusqlite` as the default would better align the SQLite stack with LanceDB's async runtime.
- **Filename Contradicts Intent:** The user explicitly requested to frame this as a sibling implementation/rewrite and *not* a migration plan. However, the documents are named `RustMigration.md` and `RustMigrationTasks.md`. Renaming them to `RustRewrite.md` or `RustSibling.md` would better reflect the intended psychological framing.
- **Chunk Identity and Hashes:** The document proposes `chunk identity` based on "source hash + chunk index + chunk content hash". If the `Chunker` trait implementation changes in the future, the chunk hashes will change, effectively duplicating the file on next ingest unless the system explicitly invalidates the old chunks for that file hash first. 

Missing Tests

- **Vector Search Parity Fuzziness:** The test plan mandates using Python Chroma outputs as "golden fixtures" for search result parity. However, Rust's `fastembed` and Python's Chroma embedding functions may have minor tokenization or floating-point precision differences, even when using the same ONNX model. This will inevitably cause rank-swapping or slight score differences in top-K results. The test plan must account for fuzzy matching, distance tolerances, or set-overlap metrics rather than exact snapshot parity for retrieval tests.
- **Cross-Database Consistency Tests:** There are no tests defined for simulating crashes during the dual-write process to ensure the system can recover from an inconsistent state between LanceDB and SQLite.

Strong Concerns

- **Over-investment in Parity:** The task plan requires a massive upfront investment in capturing and asserting against Python golden fixtures ("No production module starts without..."). Given that the intent is to build a "sibling implementation" and the Python app has acknowledged "quirks," strictly enforcing bug-for-bug parity via snapshots might severely bottleneck the Rust rewrite. The strict TDD parity rule might force you to replicate Python's heuristic flaws just to get the tests to pass.
- **MCP Crate Maturity:** The document suggests evaluating `rmcp`. The Rust MCP ecosystem is currently highly fragmented and fast-moving. Pinning hopes on a specific nascent crate for a core subsystem could lead to rewrite churn. The fallback criteria are good, but the risk of rolling a custom MCP server implementation should be explicitly acknowledged.

Confidence

High. The documents successfully fulfill all of the requester's criteria. They provide a thorough, highly detailed, and pragmatic technical plan for the Rust rewrite. The deep-dive into e2-micro constraints and explicit embedding strategies is excellent. The critiques above are standard architectural guardrails for this type of system design.
```
