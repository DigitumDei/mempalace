## Executive Summary

This branch adds `docs/RustMigration.md` and `docs/RustMigrationTasks.md`, providing a comprehensive subsystem-by-subsystem plan for porting mempalace from Python/ChromaDB to a Rust stack centered on LanceDB and SQLite with explicit local embeddings via fastembed. All three reviewers agreed there are no blocking issues — the documents are documentation-only, substantially complete, and meet the stated success criteria. However, reviewers identified a consistent cluster of specification gaps: the task plan under-specifies MCP tool surface parity (covering only ~3 of ~19 registered tools), omits the `compress` CLI command from Phase 9 despite inventorying it as current surface, and misrepresents normalization-time spellcheck as pure format handling. The `ready_for_pr` verdict reflects the absence of merge-blocking defects while acknowledging that these gaps should be addressed to make the document reliably actionable as a porting guide.

- Spellcheck mutation during normalization not described (docs/RustMigration.md): normalize.py:227 applies `spellcheck_user_text` to user turns before storage, and convo_miner.py:303 ingests that output directly. The migration doc frames normalize.py as format handling and canonical transcript formation only, which contradicts the actual parity requirement for 'verbatim' storage.
- Omitted Python modules from architecture inventory (docs/RustMigration.md): `room_detector_local.py`, `spellcheck.py`, and `split_mega_files.py` are absent from the 'Current Python Architecture' module listing. `room_detector_local.py` is particularly material — the document assigns room routing to miner.py, but a dedicated module handles it.
- `DrawerRecord.filed_at` typed as `String` contradicts document's own date guidance (docs/RustMigration.md): The Knowledge Graph section recommends using `time` or `chrono` for dates, but the `DrawerRecord` example struct uses a bare `String` for `filed_at`. This internal inconsistency will produce schema debt immediately.
- Dual-write consistency hazard has no specified recovery protocol (docs/RustMigration.md): LanceDB and SQLite share no transaction coordinator. An ingest crash can produce ghost vectors (LanceDB written, SQLite manifest not updated) or phantom completions (SQLite updated, LanceDB write incomplete). The document discusses manifests and idempotency but does not specify a concrete crash-recovery rule (e.g., SQLite as source of truth for pruning orphaned LanceDB entries on startup).
- `Chunker` trait references undefined `SourceDocument` and `Chunk` types (docs/RustMigration.md): The code example defines `fn chunk(&self, input: &SourceDocument) -> Vec<Chunk>` but neither `SourceDocument` nor `Chunk` are defined anywhere in the document. - Retrieval-golden strategy conflates exact ranking parity with format/filter parity (docs/RustMigrationTasks.md): The Python benchmark baseline uses Chroma's default embedding path (longmemeval_bench.py:99). Phase 3 switches Rust to fastembed. Same model name does not guarantee identical float vector rankings across ONNX runtimes. Exact goldens are appropriate for formatting and filter semantics; retrieval quality should be gated by benchmark thresholds and tolerant parity checks.

## Disputed Issues

- `EmbeddingProvider` trait using `anyhow::Result`: Gemini and Claude consider this a real Rust design defect for a guide meant to drive implementation; Codex considers it a style/preference issue not grounded in the document's stated goals.
- e2-micro cold-start memory pressure for ONNX + fastembed: Gemini flags this as a real deployment risk; Codex finds it unsupported without cited memory budgets or benchmarks.

## Outstanding Concerns

- The dual-write recovery protocol is the most architecturally significant unresolved gap. The document should specify a canonical consistency rule (e.g., SQLite `ingest_files` status as the authoritative source of truth; on startup, prune LanceDB rows not recorded as committed in SQLite) so that implementors have a concrete contract to test against.
- The retrieval-golden approach needs an explicit policy distinguishing exact-match goldens (formatting, filter semantics) from threshold-based parity checks (ranking quality), because Chroma and fastembed will not produce bit-identical float vectors even with the same model weights.
