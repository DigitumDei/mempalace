## Executive Summary

This branch implements Phase 4 of the Rust ingest pipeline, delivering project discovery, ignore-rule filtering, chunking, conversation export parsing, and storage writes with incremental reindexing support. However, the `source_key` scoping fix was applied only at the SQLite manifest layer: `replace_source_drawers` still queries by `source_file` alone without a wing or mode constraint, so ingesting a file in one context silently deletes drawers belonging to the same relative filename in every other wing or extraction mode — directly violating the stated success criteria. Compounding this, `drawer_id` omits `extract_mode` and root scope, causing ID collisions that silently overwrite drawers across scoped ingests, and migration `0002` writes cache keys in a format incompatible with runtime lookups, breaking the incremental cache for any upgraded database. The branch must be revised to address these data-loss and idempotency failures before it can be merged.

## Blocking Issues

- Cross-context drawer deletion in replace_source_drawers (mempalace-rs/crates/mempalace-ingest/src/lib.rs): The stale-drawer query uses DrawerFilter { source_file: Some(...), ..DrawerFilter::default() } with no wing or mode constraint. Any drawer from another wing sharing the same relative source_file string will not appear in the new batch's ID set and will be silently deleted. This is a data-loss bug on any multi-wing or multi-mode installation.
- drawer_id omits extract_mode and root scope, causing silent overwrites (mempalace-rs/crates/mempalace-ingest/src/lib.rs): drawer_id is constructed from wing, room, source_hash, and chunk_index only. Two scoped ingests of the same relative file in the same wing (e.g., Exchange vs. General modes) that route to the same room will produce identical IDs at chunk_index 0 and overwrite each other at the Lance layer before the stale-delete path even runs.
- Migration 0002 writes cache keys incompatible with runtime lookups (mempalace-rs/crates/mempalace-storage/src/sqlite.rs): Migration 0002 backfills source_key as ingest_kind || ':' || source_file, but the runtime source_key() function generates ingest_kind:wing[:mode]:root_hash:relative_path. These formats never match, so every row migrated by 0002 becomes permanently unreachable to get_ingested_file(), forcing a full re-ingest after any database upgrade and breaking the PR's central correctness claim.
- Orphaned drawers when a file shrinks to zero chunks (mempalace-rs/crates/mempalace-ingest/src/lib.rs): When a previously-ingested file produces an empty chunk vector (too short or empty), the ingest loop executes `continue` before calling replace_source_drawers. The old drawers remain in storage indefinitely and the manifest record retains the stale content hash, so the pipeline discovers the file as changed on every run but never cleans up the orphaned drawers.

## Non-Blocking Issues

- drawers_written undercounted by 1 per file in ingest_project (mempalace-rs/crates/mempalace-ingest/src/lib.rs): The telemetry path calls .saturating_sub(1) on the drawer count with no documented rationale. ingest_conversations does not apply the same subtraction. The metric is consistently wrong for project files and tests only assert > 0, so this goes undetected.
- is_dir parameter unused in Directory ignore rule matching (mempalace-rs/crates/mempalace-ingest/src/lib.rs): IgnoreRuleKind::Directory matches path parts without ever consulting the is_dir flag. A regular file named node_modules (or any DEFAULT_SKIP_DIRS entry) will be incorrectly filtered. Extension rules do check !is_dir, making this an inconsistency.
- Non-deterministic room selection on tied scores in detect_project_room (mempalace-rs/crates/mempalace-ingest/src/lib.rs): Room scores are collected into a HashMap before max_by_key is called. Rust's HashMap iteration order is randomized per execution, so tied scores yield a non-deterministic room choice, causing drawer ID churn across idempotent reruns.
- drawers_written telemetry query uses unscoped DrawerFilter (mempalace-rs/crates/mempalace-ingest/src/lib.rs): The database re-query used to compute drawers_written in ingest_project uses DrawerFilter { source_file: ..., ..default() } without a wing filter, inflating the count with drawers from other wings. It is also an unnecessary round-trip since the count could be derived from the already-held source_drawers vec.
- Synchronous embed() call blocks the async executor (mempalace-rs/crates/mempalace-ingest/src/lib.rs): build_drawers calls provider.embed() synchronously from within async ingest functions. A network-backed embedding provider will block a Tokio thread. The FakeEmbeddingProvider used in tests is instant, so this is not detected by CI.
- Directory routing heuristic too loose (room_name.contains(part)) (mempalace-rs/crates/mempalace-ingest/src/lib.rs): detect_project_room uses room_name.contains(part) for directory-to-room mapping. A single-character path component such as s/ or a/ will match the first room whose name contains that letter, causing systematic misrouting for short directory names.

### Gemini (gemini / gemini-3.1-pro-preview)

```text
Blocking Issues

1. **Data Loss on Incremental Ingest Across Contexts**
In `replace_source_drawers`, `existing` drawers are queried using `DrawerFilter { source_file: Some(source_file.to_owned()), ..DrawerFilter::default() }`. This queries across *all* wings and extraction modes because it lacks a `wing` or `source_key` filter. Since `stale` drawers are defined as any existing IDs not present in the new batch, ingesting a file in one wing (or mode) will unconditionally delete all drawers for any file with the identical relative path in every other wing. This actively destroys data and violates the success criteria.

2. **DrawerId Collisions Across Extraction Modes**
The `drawer_id` generator builds IDs using `wing`, `room`, `source_hash`, and `chunk_index`. It does not factor in the `extract_mode`. If a conversation is ingested in both `Exchange` and `General` modes into the same wing, both modes start their `chunk_index` at `0`. If their chunks happen to route to the same room (e.g., `general`), they will generate identical IDs and overwrite each other in storage, causing silent data loss.

3. **Orphaned Search Data on File Truncation**
In `ingest_project` and `ingest_conversations`, if a file shrinks below the minimum chunk size or becomes empty, the chunking functions return an empty vector. The ingest loop then executes `if chunks.is_empty() { continue; }`. This completely skips `replace_source_drawers`. The old drawers remain in the database indefinitely, and the file's manifest record is never updated, causing the pipeline to indefinitely re-process the empty file.

Strong Concerns

1. **Inefficient and Buggy Telemetry**
To calculate `drawers_written`, `ingest_project` makes a round-trip database query for all drawers matching the `source_file`, takes the length, and calls `.saturating_sub(1)`. Subtracting 1 makes no mathematical sense here, and re-querying the database inside the loop is extremely inefficient. Furthermore, because of the missing `wing` filter, it inflates telemetry with chunks from other projects. It should just be `summary.drawers_written += source_drawers.len();` before moving the vector.

2. **Non-Deterministic Project Room Routing**
In `detect_project_room`, room scores are inserted into a `HashMap` before `max_by_key` is called. Rust's `HashMap` iteration order is randomized per execution. In the event of a tie score between two rooms, the selected room will be non-deterministic, causing the `DrawerId` to churn endlessly on idempotent reruns and violating the "deterministic chunk id generation" requirement. The iteration should occur directly over the `rooms` slice instead.

3. **Dangerously Loose Directory Routing Heuristics**
In `detect_project_room`, directory mapping uses `room_name.contains(part)`. This means if the project contains a single-letter directory like `a/` or `s/`, the code will instantly route those files to the first room whose name contains that letter (e.g., routing `s/` to `services`). This is too fragile and will cause widespread misrouting.

Non-Blocking Issues

1. **Subdirectory Ignore Files Are Ignored**
`IgnoreMatcher::load` only checks for `.gitignore` and `.mempalaceignore` at the absolute root of the project. Nested ignore files in subdirectories are silently bypassed, deviating from standard git behavior.

2. **Substring Matching in Conversation Room Detection**
`detect_conversation_room` uses `.contains(**keyword)` to score conversation text. This leads to false positives (e.g., the keyword `"api"` will incorrectly trigger a match on the word `"rapid"`, and `"plan"` matches `"explain"`). 

3. **Room Config Changes Do Not Trigger Re-ingestion**
The pipeline skips unchanged files strictly based on their `content_hash`. If a user modifies their `mempalace.yaml` to change room routing rules, unchanged files will not be re-routed. This is standard behavior for caching, but the limitation should be clearly documented.

4. **Missing Golden Files Usage**
The checklist claims "Add normalization golden tests", but tests like `normalizes_claude_json_and_spellchecks_user_turns` use inline JSON strings rather than explicitly loading and asserting against the test files provided in `tests/fixtures/phase0/goldens/`. 

Missing Tests

1. **Cross-Wing Survival Test:** The existing `ingests_conversation_fixture_in_both_modes` test verifies that both extraction modes complete successfully, but it fails to assert that the chunks from the first run still exist after the second run. Adding this assertion would immediately catch the global `source_file` deletion bug.
2. **File Emptied Test:** There is no test that asserts the system's behavior when modifying a previously ingested, chunk-producing file to be an empty file. This test is necessary to expose the orphaned search data bug.
3. **Subdirectory Gitignore Test:** No test covers files excluded by a `.gitignore` located inside a subdirectory.

Confidence

High. The core logic of the Phase 4 pipeline is structurally complete, but it relies on an improperly scoped query for updating incrementally ingested files. The resulting data deletion and collision bugs are critical and block this PR from fulfilling its success criteria safely.
```
